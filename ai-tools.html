Using AI Tools Well

AI assistants have changed how code gets written. They’ll also change whether you actually grow — for better or for worse. Used well, they amplify a curious engineer. Used badly, they quietly hollow one out. This chapter is about staying on the right side of that line.

01 · Context

Why This Matters Now

AI coding assistants — Copilot, Cursor, Claude Code, ChatGPT, the next one — are now part of the daily loop for most engineers. This isn’t a trend you can opt out of and still be effective. It also isn’t a cheat code. The engineers who thrive with these tools do something specific with them; the ones who don’t slowly lose the habits that made them capable in the first place.

Key Insight

AI will produce code that compiles, passes local checks, and looks reasonable — even when it’s subtly wrong. Your job hasn’t changed: you are still responsible for the code you commit. The tools changed. The accountability didn’t.

What changes, what doesn’t

ChangesDoesn’t change
How fast you produce a first draftWhether you understand what you’re shipping
How quickly you scaffold boilerplateWhether the design is right for the problem
How easily you try three approachesWhether you can defend the one you kept
How much syntax you have to rememberWhether you can debug when it goes wrong
How quickly you explore an unfamiliar APIWhether you built a mental model you can reuse

The right column is where your craft lives. If the left column expands but the right doesn’t, you’re getting faster at producing code you can’t own.

02 · The Trap

The Curiosity Tax

Every time you accept AI-generated code without understanding it, you pay a small tax on your future ability to think about that code. One instance is harmless. A year of it is career-shaping — and not in the direction you want.

How it happens

You’re tired. The story is due. The AI produces something that works. You ship it. Tomorrow the same thing, in a different file. A month later you’re touching a part of the code the AI wrote and you don’t recognise any of it. You can’t change it without asking the AI again. You haven’t been writing code — you’ve been approving it.

⚠

The slow erosion: an engineer with the curiosity tax paid for a year looks fine in calm weeks. They slow down the moment something unusual happens — a novel bug, a library they can’t look up, a production incident at 2am, a design discussion that asks why the code does what it does. Those are the weeks that define a career. Those are the weeks where the tax comes due.

The compounding happens both ways

The good news: the inverse also compounds. Every time you treat the AI’s output as a starting point — to question, learn from, verify, refine — you build the exact muscles you would have built writing it yourself, with a faster loop. The tool is the same. The stance is what changes the outcome.

Common pitfall — Approver engineer

Read AI output quickly. Skim for red flags. Commit. Repeat. A year later you’re as fast as anyone — and stuck the moment you’re off the tool.

Better — Author engineer using AI

The AI drafts. You still author. You read every line, ask “why this over X?”, reshape for your codebase’s style, and add the test you’d have written regardless. Your growth doesn’t pause when the model does.

03 · Triage

Accept, Verify, or Reject

For every chunk of AI output, you’re making one of three decisions. Most engineers only notice the “accept” decision — and that’s the one that silently hurts them. Make the triage conscious.

The AVR Triage (Accept / Verify / Reject)

  1. Read every line the AI wrote. No skimming. You can read faster than you can write, but not faster than the AI can make mistakes.
  2. Ask: do I understand what each line does, and why? If yes — it’s a candidate for Accept.
  3. If any line is unclear — either ask the AI to explain it, or look it up in the real docs. Verify until you understand it. Don’t commit what you can’t explain.
  4. Ask: does it fit this codebase? Match style, match idioms, use the existing helpers. If not — Reject or reshape. “Works” is not enough.
  5. Ask: would I have written it this way? Not because your instinct is always right — but because this is the moment to compare and learn. “The AI did X; I’d have done Y; which is better here?” is the question that builds you.
  6. Add what’s missing. The AI is bad at boundaries: the edge case, the error path, the obvious test. That’s your layer.

A scenario: the plausible wrong function

🔎

You ask the AI for a function to chunk a list into batches of n. It produces a clean-looking 8-line function. You accept it. Three weeks later a batch job silently drops the last partial batch when the list length isn’t divisible by n. The bug sits in production for a month.

What AVR catches: in step 2 you’d have asked “what happens when length isn’t a multiple of n?”. In step 6 you’d have added a test for that exact case. Neither step takes more than two minutes. Both would have prevented the bug.

Key Insight

The AI will confidently produce code that is subtly wrong far more often than code that obviously is. Your verification energy should match: generous on the “looks fine” output, not just the “looks weird” output.

04 · Prompting

Prompting Is a Craft Skill

How you ask shapes what you get. A sloppy prompt produces sloppy code — confidently. A precise prompt produces something you can actually use. Prompting isn’t mystical; it’s specification, and specification is already a skill engineers respect.

The CCEE Prompt Pattern (Context, Constraint, Example, Exit)

  1. Context. What is this code part of? Language, framework, style conventions, existing helpers. “We’re in a Spring Boot 3.4 service, using JUnit 5 and AssertJ. Records are the norm for DTOs.”
  2. Constraint. What must the answer satisfy? “Pure function, no I/O, handles empty input by returning an empty list, handles null by throwing IllegalArgumentException.”
  3. Example. If there’s a similar function in the codebase, paste it. The model pattern-matches; give it the right pattern.
  4. Exit. How will you know it’s right? “Here are the cases I care about: empty, one element, many, exact multiple of batch size, non-multiple.”

Why this beats “write me a function that…”

A bare “write me a function that batches a list” is under-specified. The AI will fill in the blanks — with a reasonable-looking guess that may not match your codebase, may miss your edge cases, and may use a different convention than the file next to it. Every line of context you add narrows the space the AI invents in. That narrowing is the craft.

Common pitfall — The one-liner prompt

“Write a function that sorts users by last activity.” No idea of the data shape, no idea of null handling, no idea of the codebase’s sort conventions. You get something generic and spend 20 minutes reshaping it.

Better — The specified prompt

“Given a List<User> (see attached record), return a new list sorted by lastActivityAt descending, nulls last, stable sort. We use List.copyOf and streams; no external libraries.” You get something you can actually use.

Common pitfall — Accepting the first answer

The model offered something. You take it. You don’t realise there were three better designs it would have suggested if you’d asked.

Better — Ask for alternatives, then choose

“Show me three different approaches to this, with the tradeoffs.” Now you’re designing, not accepting. You also learn — the tradeoff list is free education.

Prompting to learn, not just to produce

The most underused mode: “explain this code to me, line by line, assuming I know Java but don’t know this library.” “What would a senior engineer have pushed back on in this diff?” “What’s the design decision I’m not seeing?” Treat the AI as a tutor you can keep interrupting. That’s a learning loop that didn’t exist five years ago.

05 · Verification

The Verification Habit

Read AI code the way you’d read a stranger’s pull request — not your own. You wrote none of it. You owe it every question you’d owe any unfamiliar code.

The Five Reads

  1. Intent read. Does this actually solve the problem you asked for, or a slightly different one?
  2. Correctness read. Are the logical branches right? Off-by-one, null handling, edge cases, error paths.
  3. Fit read. Does it match the codebase’s style, use existing helpers, respect the module’s conventions?
  4. Security read. Any user input going somewhere unsafe? Any secret, token, or PII in logs or error messages? Any obvious injection surface?
  5. Test read. If I delete this and re-run the tests, would the tests tell me?

A quiet rule: write the test yourself

Let the AI draft the implementation. Write the tests yourself — at least some of them. Why: the AI tends to write tests that pass because it wrote them for its own implementation. Tests you design from the requirement, independent of the code, are the ones that catch the AI being subtly wrong.

💡

A good split: AI drafts the code. You write at least one “hostile” test — empty input, malformed input, concurrent use, boundary — before you ask it for more tests. That one test catches more bugs than the five the AI adds for you.

Hallucinations to watch for

Imaginary APIs

The AI invents a method that doesn’t exist on the library you’re using — often a plausibly-named one. Check the real docs.

Stale patterns

The model was trained on years of code. It sometimes writes 2018-style patterns instead of what’s idiomatic today. Cross-check with your codebase’s newer code.

Silent fallbacks

It catches exceptions and returns a default without telling you. Failures become invisible. Read the catch blocks carefully.

“Works on the happy path” tests

AI-generated tests lean toward the happy path. The test file looks thorough — it tests what the code already does right. It rarely tests what it’s likely to get wrong.

06 · Patterns

Where AI Helps, Where It Quietly Hurts

The tool is the same in every situation; the outcomes aren’t. Building an honest map of where AI helps you and where it hurts you is the difference between using it well and pretending it always works.

Where AI genuinely helps

Boilerplate & scaffolding

DTOs, repetitive CRUD, config files, parser shells. Work you’ve done before and don’t learn from repeating.

Exploring an unfamiliar API

“Show me three ways to use this library.” Faster than reading three tutorials. Verify the one you pick against the real docs.

Test data and fixtures

Generating realistic test data, edge cases, parameter sets. Often faster and more thorough than writing them by hand.

Writing comments / docs

You already know what the code does; the AI can phrase it. Always re-read — it will sometimes describe what it thinks the code does rather than what it actually does.

Rubber-ducking at 2am

Explaining a bug to the AI is a disciplined rubber-duck exercise. Often you solve it during the explanation. The AI’s answer is a bonus.

Translating between languages

Porting a small piece of logic, converting tests between frameworks, bridging code and SQL. Faster than manual translation, easier to verify.

Where AI quietly hurts

Learning the language or framework you use every day

If you never read the real docs, you never build the mental model. AI lets you skate along without deepening — which is fine for one-offs and disastrous for your core stack.

Debugging a real production issue

Complex bugs require theory, evidence, and a mental model of the running system. AI can suggest; it can’t investigate. The disciplined debugger in your team is the one who owns the incident.

Design decisions

“Should we use a queue or a cron job?” depends on context the AI doesn’t have: team size, ops capacity, existing infrastructure, failure modes that matter. Use AI to enumerate options; make the decision yourself.

Code review of others’ MRs

If you outsource your review thinking to the AI, you stop noticing the team’s patterns, stop catching the real trade-offs, and deliver review comments that feel generic. Your teammates notice.

Sensitive code paths

Auth, billing, migrations, anything touching PII. Confident AI output in these paths is the most dangerous shape it takes. Treat AI suggestions here as the first draft, not the final one.

Anything you can’t explain in the review

If a reviewer asks “why this approach?” and your honest answer is “that’s what it gave me,” you shouldn’t have committed it.

07 · Growth

Guarding Your Own Growth

The biggest career risk of AI tools isn’t that they’ll replace you. It’s that they’ll make you feel productive while quietly pausing your growth. Set the guardrails yourself — nobody else will set them for you.

The Growth Guardrails

  1. One deep thing per week without the AI. Pick one task — a bug, a feature, a refactor — and do it fully yourself. No autocomplete, no prompt. You’re practising the craft.
  2. Read docs before asking. When you need a new library or API, spend ten minutes in the real docs first. Then compare what the AI gives you.
  3. Explain what you shipped. At the end of the day, could you walk a teammate through every line of what you committed? If no, you committed too early.
  4. Teach it back. Once a month, write a short note or demo to the team explaining something you used AI to build. Teaching surfaces what you actually understood.
  5. Track your “stuck without it” moments. If the AI is down or slow and you freeze, that’s a skill gap the tool was hiding. Write it down. It’s your curriculum.

Key Insight

The goal is not to use less AI. The goal is to use AI without outsourcing the thinking. Those are different things. One makes you faster; the other makes you replaceable.

🌱

Why this compounds: engineers who use AI tools as amplifiers keep growing. Engineers who use them as substitutes hit a ceiling they can’t see. Sustained practice as an amplifier beats equivalent practice as an approver — and the difference shows the first week the tool is unavailable, or the first production incident that needs a real investigator.

08 · Team

AI on a Team, Not Just on You

How you use AI affects your teammates — in review, in incidents, in pairing, in the codebase they’ll maintain. The individual habits above are necessary. So are the team habits below.

Patterns & anti-patterns

Common pitfall — Silent AI use in review

You shipped AI-generated code. The reviewer doesn’t know. They assume you can explain every choice. You can’t. The review stalls or drifts into confusion.

Better — Flag where it matters

In the MR description: “The batch retry logic was drafted with AI; I’ve verified cases X, Y, Z — please pay extra attention to the error path.” Doesn’t mean apologising for using the tool — it means pointing reviewers at the parts most likely to hide a mistake.

Common pitfall — Secrets or private code in prompts

Pasting real API keys, customer data, or proprietary code into tools without checking the org’s policy. Once pasted, it’s out.

Better — Follow the policy, sanitise by default

Know your org’s AI policy. Use company-approved tools for sensitive code. Sanitise prompts of secrets, PII, and customer identifiers — the same way you’d sanitise a public gist.

Common pitfall — Review by AI

You paste a teammate’s MR into the AI and post its comments as your review. The comments feel generic because they are. Your teammate learns nothing specific about their code.

Better — AI as a second pair of eyes, you as the reviewer

Use the AI to surface questions you might have missed. Then do your own review — with the context you have that the AI doesn’t.

Common pitfall — The “it works on my machine” trap, AI edition

The AI produced code that passed in your environment. You didn’t check why it might not elsewhere. Prod surfaces the difference.

Better — Run it where it will run

Test in an environment that matches production: same versions, same data shape, same configuration. The AI can’t do this for you.

Dos & Don’ts

DoDon’t
Read every line the AI wrote before committingSkim for obvious red flags and ship
Provide context, constraints, and examples in your promptExpect the one-liner to produce what you need
Write at least one hostile test yourselfAccept the AI’s test suite as sufficient
Pick one task a week to do without AI assistanceLet autocomplete become your default typing tool
Flag AI-drafted sections in MR descriptionsLet reviewers assume you wrote every line yourself
Treat AI as a tutor — ask for explanations, alternatives, critiquesTreat it as an oracle whose output you approve
🔮

Why this matters: the team’s code is a shared artifact. The AI doesn’t know your team’s conventions, disagreements, or scars. You do. What you commit is what your teammates will maintain at 2am in six months. Own it like you wrote it — because as far as the codebase is concerned, you did.

Further reading

  • The Pragmatic Programmer (20th anniversary ed.) — Hunt & Thomas. The chapters on tool mastery and deliberate practice apply directly to how you hold an AI assistant.
  • Simon Willison’s blog (simonwillison.net) — one of the clearest, most honest ongoing writeups of what AI tools can and can’t do, updated weekly.
  • Addy Osmani — writing on AI-assisted engineering workflows. Good on habits, honest about failure modes.
  • Anthropic’s “Building effective agents” — worth reading once to understand how these tools actually construct answers.

One thing to try this week

  • Pick one task and do it entirely without AI help. Notice what feels harder — that’s your growth edge.
  • Next time you reach for an AI prompt, write the Context/Constraint/Example/Exit version instead of the one-liner.
  • Before you commit AI-drafted code, write one hostile test yourself that the AI didn’t suggest.
  • In your next MR that used AI, flag the AI-drafted sections in the description so reviewers know where to look hardest.
  • Use the AI as a tutor once: paste a piece of code you didn’t write and ask “explain this, and what would a senior engineer push back on?”
