01 · Readability

Readability First

Code is read far more than it is written. Every line you write will be read — by a teammate, a reviewer, a future-you — dozens of times over its life. Optimising for the writer is cheap at the keyboard and expensive forever after.

Why readability beats cleverness

A clever one-liner feels satisfying. It also costs every subsequent reader 30 seconds of “wait, what is this doing?” Over a codebase’s life, that adds up to hours. The boring, obvious version costs you zero to write and zero to read.

Key Insight

The primary audience of your code is the next engineer who reads it — not the compiler, and not you at the moment of writing. Optimise for them.

The readability moves

The Readability Checklist

  1. Short functions. If it doesn’t fit on one screen, it’s probably doing more than one thing.
  2. Meaningful names. calculateTax beats calc. isEligibleForRefund beats flag.
  3. One level of abstraction per function. Either high-level (orchestration) or low-level (details) — not both mixed.
  4. Early returns over deep nesting. Three levels of if is a code smell.
  5. Comments explain why, not what. The code shows what. Only write a comment if the reason isn’t obvious from the code.
  6. Dead code deleted. Commented-out code is confusion. Git remembers; your file doesn’t have to.

Scenario: the clever filter

🔎

You’re reviewing a PR. You hit:

const result = users.filter(u => u.r?.a || (u.r?.p && new Date(u.r.p) > new Date())).map(u => u.e);

You stop. You read it three times. What’s r? What does a mean? Is p a date or a boolean? You scroll up to find the type. You scroll back. Thirty seconds gone — on one line.

The readable version:

const activeEmails = users
  .filter(user => hasActiveRole(user))
  .map(user => user.email);

function hasActiveRole(user) {
  if (!user.role) return false;
  if (user.role.isAdmin) return true;
  return user.role.permissionsValidUntil &&
         new Date(user.role.permissionsValidUntil) > new Date();
}

Same behaviour. Reads like English. Multiply by every reader forever — that’s the return on the extra five lines.

Dos & Don’ts

DoDon’t
Extract confusing expressions into named variables or functionsWrite expressions that require comments to understand
Use early returns to flatten nestingWrite pyramid-of-doom if chains
Write code a teammate could understand without youWrite code only you can maintain
Delete dead code now, not “later”Leave commented-out blocks “just in case”
Break a 40-line function into three 12-line onesKeep adding “just one more case” to a long function
02 · Naming

Naming as a Design Decision

Naming is not labelling. A good name is a design decision — it encodes intent, signals shape, and prevents confusion. A bad name makes the reader scroll, the reviewer ask, and the bug hide. If you only build one habit from this chapter, build this one.

Why naming is design

Consider a function called process(data). What does it do? You have to read the body to know. The name has given you nothing. Now consider redactPiiFromPayload(payload). You know what goes in, what comes out, and why. You can review the call site without reading the function. The name is the interface.

The Naming Moves

  1. Name for intent, not implementation. findEligibleOrders() beats queryOrdersTableWithJoin().
  2. Use the domain language. If the business calls it “customer”, don’t call it user in code. Align vocab.
  3. Say what it is, not what it isn’t. isActive beats isNotDisabled.
  4. Match pairs. open/close, start/stop, create/delete. Not open/destroy.
  5. Avoid vague words. data, info, manager, util, helper, handle — all signs you haven’t thought of the actual thing.
  6. Length proportional to scope. A loop counter can be i. A module-level constant shouldn’t.

The named-argument trick

You’re reading:

createUser("alice@example.com", true, true, false);

What do the booleans mean? You have to read the signature. Every reader does. Forever.

Languages that support named args (Kotlin, Python, Swift, C#) let you write:

createUser(email = "alice@example.com", isAdmin = true, sendWelcome = true, forceSync = false);

In languages that don’t (Java pre-records, JS without objects), pass an options object or split the function. The cost is a few characters; the benefit is every reader understands the call without scrolling.

Common pitfall — Generic wrapper names

DataManager, UserHelper, OrderUtils. The names signal “I couldn’t think of what this actually does.” The classes accumulate unrelated functions.

Better — Specific, intent-led names

OrderPricingCalculator, UserInvitationSender, OrderCsvExporter. Each name constrains what belongs inside, preventing the catch-all problem.

Common pitfall — Cryptic abbreviations

usrOrdCnt. You saved two keystrokes. Every reader pays mental cost forever.

Better — Spell it out

userOrderCount. Autocomplete makes the keystroke cost zero. The clarity is permanent.

Common pitfall — Negative names

if (!isNotEmpty). The double negative is a tripwire. Readers have to mentally flip it.

Better — Positive names

if (isEmpty), if (!isEmpty). Your brain can parse the second instantly.

Common pitfall — Inconsistent vocabulary

Some files say user, others say account, others say customer — all referring to the same thing. Readers have to translate on the fly.

Better — One term per concept, applied everywhere

Pick one: customer. Use it in models, services, API responses, UI copy. The whole system speaks one language.

🎯

Why this matters: of all the micro-decisions in a codebase, naming has the highest leverage. One good name saves every future reader seconds. One bad name costs them — quietly, every time they read it. There is no other habit in coding where small effort compounds as fast.

03 · Refactoring

Refactor-as-You-Go

Refactoring is not a separate ticket you do later. It’s something you do inside the ticket you’re already on, to leave the code better than you found it. Boy Scout rule: always leave the campsite cleaner than you found it.

Small, continuous, in-the-moment

Big-bang refactors rarely happen. They get scheduled, then descoped, then forgotten. Small refactors happen because they cost nothing: you’re already reading the code, already editing it, already testing it. Rename this variable. Extract that block. Delete this dead branch. Ten minutes inside each story compounds into a codebase that gets better instead of worse.

Refactor-as-You-Go Moves

  1. Rename when you spot a bad name. Your IDE does it safely in two seconds.
  2. Extract a function when a block is doing its own thing. Especially if it has a clear name waiting to be given.
  3. Delete when you’re sure. Dead branches, dead variables, dead comments. Git remembers.
  4. Split when a file grew past “I can’t find things easily.” Cohesion over convenience.
  5. Only touch what you’re there for — resist scope creep. Small PRs.

The “while I was here” trap

The opposite failure mode: you open a file to change one line, and end up refactoring the whole class. Now your MR is 600 lines long, the reviewer can’t tell the bug fix from the cleanup, and the cleanup introduces its own small bug. Two hours of your time and two hours of the reviewer’s, for no good reason.

⚠

The rule: refactors you can do in 5 minutes and explain in one line — do them in the same MR. Refactors bigger than that — open a follow-up ticket, mention it in the current MR (“I noticed X, tracked as TICK-xyz”), and keep your current MR scoped.

When NOT to refactor

Refactor-as-you-go doesn’t mean refactor-always. Skip it when:

  • There are no tests. Add tests first (or accept the risk consciously).
  • You’re mid-incident. Fix the bug, ship, clean up tomorrow.
  • The code is about to be deleted. Don’t polish what’s leaving.
  • You don’t fully understand it. Read first; refactor only when you can predict the behaviour you’re preserving.
Common pitfall — Never refactoring

“Not my ticket.” Every month the codebase gets a little worse. Nobody notices the day it tipped from manageable to painful.

Better — Small improvements inside every ticket

One rename, one extraction, one delete. Ten minutes. Ship. Next ticket, same thing. Compounds over years.

Common pitfall — The giant rewrite MR

“I noticed X and cleaned it up.” 900 lines. Reviewer sighs. Gets stuck in review for a week.

Better — Follow-up ticket with a scoped MR

Open it separately. Small, focused, reviewable. Often mergeable in hours.

04 · Language Fluency

Using Modern Language Features

Your language has been evolving. If you’re still writing it the way the tutorial you learned on showed you, you’re writing 2015-era code in 2026. This isn’t about being trendy — modern features exist because they solve real problems. Learning them is part of the job.

Why this matters

Modern language features tend to make code safer, shorter, and clearer. Null safety avoids whole classes of bugs. Pattern matching replaces fragile if/else chains. Records/data classes eliminate boilerplate that used to hide bugs. When you write the verbose old-style equivalent, you’re choosing the worse option on purpose — usually without realising it.

What to keep an eye on

Java

Records, sealed classes, pattern matching for switch/instanceof, text blocks, var, enhanced Optional, streams and collectors, virtual threads (21+).

TypeScript

Discriminated unions, satisfies, template literal types, const type parameters, proper never usage, strict null checks.

Python

Type hints (typing, pydantic), dataclasses, pattern matching (match), f-strings, pathlib, walrus operator where it reads well.

JavaScript

Optional chaining, nullish coalescing, array/object destructuring, async/await, proper Promise.allSettled, ES modules, top-level await.

Kotlin / Swift / Go / Rust

Each has its own idioms that shape what “good” looks like. The standard library is your first teacher — then style guides and patterns blog posts.

SQL

Window functions, CTEs, ON CONFLICT / upserts, JSON operations if your DB supports them. Stop writing application-side loops for things SQL can do in one statement.

Scenario: the verbose null check

🛠

Old Java:

if (order != null) {
  if (order.getCustomer() != null) {
    if (order.getCustomer().getAddress() != null) {
      return order.getCustomer().getAddress().getCity();
    }
  }
}
return null;

Modern, Optional-aware Java (or equivalent in any language with null-safe navigation):

return Optional.ofNullable(order)
    .map(Order::getCustomer)
    .map(Customer::getAddress)
    .map(Address::getCity)
    .orElse(null);

Same behaviour, half the lines, half the mental load, impossible to accidentally call through a null. The older code isn’t “more explicit” — it’s just older.

How to learn them — without boiling the ocean

The Feature Learning Loop

  1. Once a quarter, skim the release notes for the language versions your team uses.
  2. Pick one feature that looks relevant to what you write.
  3. Find three places in your codebase where it would help. Use them in your next MR.
  4. Share what you learned. A short Slack note, a lunch & learn, a PR description that shows off the feature. Your team learns with you.
Common pitfall — Writing 2015-style code in 2026

Verbose null checks where null-safe navigation exists. Raw for-loops where streams/list comprehensions exist. getX() chains where records exist.

Better — Use what the language gives you

Not dogmatically, not trendily — just: when the modern idiom is clearer and safer, choose it. Your reviewer shouldn’t have to suggest the obvious modern replacement.

Common pitfall — Hiding behind “that’s our style”

“We don’t use Optional here.” Why not? If the reason is “because we didn’t”, that’s drift, not a style choice.

Better — Question the convention, propose the upgrade

See Questioning & Proposing — Challenging Anti-Patterns Kindly. Ask. Propose. Show an example of the cleaner approach in a small PR.

05 · Smells

Common Anti-Patterns (and What to Do Instead)

These are the smells that show up in almost every codebase, usually written by people who didn’t know they were smells. Recognise them, name them, and know the cleaner alternative. You’ll see all of these in your codebase this month — some of them in your own code.

Copy-paste programming

Smell: the same five lines repeated in four places. Change one, forget the others, introduce a bug.

Fix: extract a function. Even if it’s called once today, two months from now it’ll save you.

Magic numbers

Smell: if (age > 65) with no explanation. What’s 65? Retirement? Legal? A typo?

Fix: name it. RETIREMENT_AGE_THRESHOLD = 65. Now the reader knows.

God functions

Smell: a 300-line function doing requirements validation, data fetching, transformation, and output formatting.

Fix: one level of abstraction per function. Orchestrator calls four 30-line helpers.

Primitive obsession

Smell: passing a phone number as a String, a price as a double, a user ID as a raw int. Easy to mix up.

Fix: wrap primitives in small value types (PhoneNumber, Money, UserId). The compiler now catches mix-ups.

Boolean parameters

Smell: createUser(name, email, true). What does true mean at the call site?

Fix: named args, enums (Role.ADMIN), or two distinct methods (createAdmin, createStandardUser).

Deep nesting

Smell: if { if { if { if { … } } } }. Each level costs reader attention.

Fix: early returns / guard clauses. Flatten the happy path down the left edge of the file.

Silent catches

Smell: try { … } catch (Exception e) { /* ignore */ }. A bug waiting to never be noticed.

Fix: at minimum log with context. Usually: let it bubble, handle at the boundary, or convert to a domain-meaningful exception.

Shotgun changes

Smell: adding a new field means editing 12 files in slightly different ways.

Fix: the related logic should live together. If a change always touches the same N places, those N places want to be one.

Comments that repeat the code

Smell: // increment counter above counter++. Adds nothing, rots independently.

Fix: delete. Comments explain why, not what. If the code needs a “what” comment, rename.

God class / god config

Smell: one class or config file knows everything, changed by everyone, merge conflicts every week.

Fix: split by cohesion. Things that change together stay together; things that change separately should live separately.

Abbreviated names

Smell: usr, tmp, proc, mgr. Keystrokes saved, clarity lost forever.

Fix: spell it out. user, temporaryFile, processOrder, userAccountManager.

Commented-out code

Smell: 20 lines commented out “in case we need them later.” You won’t. Nobody will trust the code around them.

Fix: delete. If you genuinely need it, it’s in git history. Trust git.

Key Insight

Recognising an anti-pattern is half the skill. Having a handful of well-known alternatives ready — so you can propose the fix in review or refactor in passing — is the other half. Both come from the naming: once you can name it (“that’s primitive obsession”), the fix is obvious.

06 · Perspective

The Six-Month Test

Before you merge anything, run it through a simple imaginative check: “Six months from now, a teammate is debugging a production issue at 10pm and they land on this code. Will they understand it?” If the honest answer is no, the code is not done yet.

Why “six months”

Six months is long enough that you won’t remember the context you had today. The clever shortcut that feels obvious now will feel opaque to future-you (and anyone else). Six months is also short enough that the code is still current — you’re not designing for some hypothetical “when the team changes” — you’re designing for next week, slightly fogged.

The Six-Month Questions

  1. Can a reader tell what this does from the names alone?
  2. Can a reader tell why this does it, either from the code or one good comment?
  3. If something goes wrong, will the error or log message help them?
  4. Are there tests that document the behaviour?
  5. If a reader wants to change one thing, is it obvious where to make the change?

If all five are “yes”, the code is ready. If not, you have a small pre-merge to-do list.

🎯

Why this matters: code that passes the six-month test ages gracefully. Code that doesn’t becomes the “nobody touches that module” module three years later — and teams lose weeks working around it. The small investment you make today at merge time is the interest-free loan that keeps paying back forever.

Further reading

  • Clean Code — Robert C. Martin. Controversial in parts but foundational; chapters on naming, functions, and comments alone are worth the read.
  • A Philosophy of Software Design — John Ousterhout. Short, sharp, deeply practical on complexity.
  • Refactoring — Martin Fowler. The catalogue of refactors and when to apply them.
  • Kevlin Henney’s talks on readability (on YouTube) — entertaining and changes how you look at code.
  • Sandi Metz, Practical Object-Oriented Design in Ruby — language-specific on the cover, universal in the lessons.

One thing to try this week

  • Rename one poorly-named variable or function in the next MR you open.
  • Find one anti-pattern from section 5 in your codebase. Either fix it or open a ticket.
  • Replace one verbose old-style idiom with a modern language feature in your next story.
  • Run the Six-Month Test on your next MR before asking for review.
  • Refactor inside a ticket — one small thing, 10 minutes max, in the same MR.
