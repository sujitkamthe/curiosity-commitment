Debugging & Reading

Bugs are a practice, not a talent. The scientific method beats intuition; reproduction beats guessing; reading the stack trace beats staring at the code. And before you can debug a codebase, you have to be able to read it.

01 · Method

The Scientific Method for Bugs

Debugging is not a personality trait. It’s a process you can learn and run deliberately. The engineers who seem to debug “fast” are almost never faster thinkers — they just follow the method more disciplined. The method is the scientific method, applied to code.

The flow

The Debug Loop

  1. Observe. What is actually happening? Exact inputs, exact outputs, exact error. Not “it’s broken” — “when X happens, the response returns 500 with message Y.”
  2. Hypothesise. What could cause this? Pick the most likely candidate. Write it down.
  3. Predict. If the hypothesis is correct, what should I see when I check Z? Make the prediction before looking.
  4. Experiment. Check Z. Either the prediction holds or it doesn’t.
  5. Narrow. If the prediction held, you’re closer. If it didn’t, you’ve eliminated a hypothesis. Go back to step 2.

Why “fast” engineers are actually disciplined

When you watch a senior engineer debug, it often looks like intuition. They tried this one thing, and it was it. What’s actually happening: they’ve run the loop many times, and their hypotheses start closer to correct because they’ve seen the shape of this bug before. You don’t have their pattern library yet. You will — by running the method explicitly for a year and building the library one bug at a time.

Key Insight

The engineer who guesses fast usually debugs slow. Random guesses feel productive and rarely are. Fifteen minutes of disciplined hypothesis testing beats two hours of “what if I try this?”

Scenario: the vanishing field

🔎

A user reports: “my address doesn’t show up in the invoice.” You look at the code. It looks right. You spend 20 minutes staring at the invoice generator.

With the method:

  • Observe: specifically, which user? When? What did the invoice actually look like? Get the real invoice open in front of you.
  • Hypothesise: either (a) the data wasn’t saved, (b) it was saved but not fetched, (c) it was fetched but not rendered.
  • Predict: if (a), the DB has no address for this user. If (b), DB has it but the fetch query filters it out. If (c), logs would show the value being passed to the template.
  • Experiment: query the DB directly. Address is there. Hypothesis (a) eliminated.
  • Narrow: now it’s (b) or (c). Log the fetched data. Is the address there? If yes — it’s (c), a rendering bug. If no — it’s (b), a fetch bug.

In ten minutes you’ve localised the bug to one layer. Then you can stare at that code.

Common pitfall — Guessing without predictions

“Let me try changing this.” You make five changes in ten minutes without knowing what each should prove. Half-fix at best; lost track of state at worst.

Better — One hypothesis, one experiment

State the hypothesis. State what you expect to see. Then run the experiment. Even if you’re wrong, you eliminated something.

Common pitfall — Staring at the code

You re-read the same function six times hoping something jumps out. Nothing does. An hour gone.

Better — Prove what’s actually happening

Run it. Log it. Inspect it. Real observations beat re-readings. Static staring is for code review, not debugging.

02 · Reproduction

Reproduce Before You Fix

A bug you can’t reproduce is a bug you can’t fix. You can only “fix” it in theory — and the next production occurrence proves you didn’t. Reproduction is the foundation of all real debugging.

Why reproduction first

If you can reproduce the bug, you can:

  • Verify your hypothesis (does fixing X make the bug go away?)
  • Confirm the fix actually works
  • Write a regression test
  • Show the bug to others without waiting for production

If you can’t reproduce, you’re flying blind. Every “fix” is a guess, and you won’t know if it worked until the bug either happens again (fix didn’t work) or doesn’t (might’ve been coincidence).

The Reproduction Ladder

  1. Local reproduction with real data. The gold standard.
  2. Local reproduction with synthetic data. Works when real data is sensitive or huge.
  3. Staging reproduction. When the bug depends on infra you don’t have locally.
  4. Capture-from-production reproduction. A failing request recorded in logs, replayed against staging.
  5. Theoretical reasoning only — last resort. Accept that the fix may not work.

Scenario: the “works on my machine” bug

⚠

A user reports a date is showing one day off. You can’t reproduce it. “Works on your machine” meets “doesn’t work on theirs.”

The reproduction path: get the user’s timezone. Get their browser/OS. Get the exact date they saw wrong. Set your local machine to their timezone (change OS timezone or run the browser with a TZ env var). Try the same input.

Often this reproduces instantly. The bug wasn’t “sometimes” — it was timezone-dependent, and your timezone happened to be on the side of the boundary that doesn’t trigger it.

If you didn’t chase reproduction, you’d “fix” the bug by adjusting a format string, ship it, and the bug would silently persist for another user in another timezone.

Common pitfall — Fixing without reproducing

You read the bug report, have a theory, push a fix. Maybe it works. Maybe it doesn’t. You won’t know for sure.

Better — Reproduce first, fix second

Even an extra 30 minutes to reproduce pays back in certainty. The fix becomes provable.

Common pitfall — Giving up when it doesn’t reproduce easily

“I can’t reproduce.” You close the ticket or mark it as flaky. The bug reappears in a week.

Better — Climb the ladder

Try different data. Try different inputs. Try their exact environment. Record a failing request and replay it. The more edge-case-y the bug, the further up the ladder you may need to go — but most bugs are reachable with effort.

03 · Reading

Reading Stack Traces Properly

A stack trace is a map. Most engineers glance at the top and close it. The full value of the trace is in reading it like a story — top to bottom, then filtering for your own code, then opening the right frame.

What a stack trace tells you

A stack trace tells you, in order: what broke, and the sequence of function calls that got there. The top of the trace is where the exception was raised. The bottom is usually where the program started. Your own code is somewhere in between — often sandwiched between framework frames and library frames.

How to Read a Trace

  1. Read the top line. The exception type and message. “NullPointerException” tells you one thing, “IllegalArgumentException: size cannot be negative” tells you a very different thing.
  2. Scan down for your code. The first frame that’s in your package/namespace. That’s the nearest point in your code to the exception.
  3. Note the line number. Look at that line.
  4. If the code looks fine, look at the frame above it. What did it pass in? What state was it in?
  5. Read the whole call chain at least once. Sometimes the bug is not at the top of the trace — it’s that the code even got to this call.

Scenario: the misleading top frame

🛠

Your stack trace shows an NPE in String.substring(beginIndex, endIndex). You stare at the String class. That’s definitely not where the bug is — it’s in the JDK.

Read down: substring was called by OrderFormatter.truncateId at line 42. There, you see order.getId().substring(0, 8). So order.getId() is null. Back up one more: truncateId was called from OrderFormatter.format, which handles any order.

The bug: a newly created order doesn’t have an ID yet when it’s passed to format — the ID is assigned later. The fix: either handle null, or don’t call format on unsaved orders.

The top of the trace (String.substring) told you nothing. The third frame down told you everything.

Common pitfall — Glancing at the top line only

“Null pointer exception.” Okay. Now you open the file and stare at the top of it.

Better — Find your frame

Scroll to the first frame in your own code. Note the line. Look at the call. That’s almost always where you start.

Common pitfall — Ignoring the exception message

The message often tells you the specific thing that went wrong — “index 5 out of range for length 3.” People skip past it.

Better — Read the message word by word

“Index 5 out of range for length 3” is a gift — it narrows the bug to an off-by-one in the indexing of a 3-element thing. Don’t skim.

Common pitfall — Trusting the filename blindly

The trace says OrderService.java:42. You open your OrderService.java — but that’s a different OrderService in a different package. Wrong file, wrong answer.

Better — Check the full package path

Jump-to-file features use the full path. Use them. Don’t hand-search by filename.

04 · Tools

Using the Debugger (Not print())

Every IDE has a debugger. Most engineers use it rarely. If you debug with print/console.log/System.out.println as your default, you’re probably spending 3–5x longer on bugs than you need to. The debugger is one of those skills where a weekend of practice pays for itself forever.

What the debugger actually gives you

Debugger Superpowers

  1. Breakpoints. Pause execution exactly where you want.
  2. Conditional breakpoints. Only pause when a condition is true. “Stop only when userId == 'alice'.” Game-changer.
  3. Step over / step into / step out. Walk through code at the level of detail you need.
  4. Watch expressions. See the value of any variable or expression live.
  5. Evaluate in the current frame. Try any expression in the running context. No recompile.
  6. Stack inspection. See every call above you and the variables at each level.

When print() is actually fine

Don’t be dogmatic. print-debugging is fine for:

  • Quick sanity checks (“is this function even getting called?”)
  • Debugging async / concurrency, where stopping the world changes the behaviour
  • Observability gaps you’re about to fix by adding real logs anyway

But the moment you’ve added three print statements and are re-running, stop: you’ve passed the break-even point where the debugger would be faster.

💡

Why this matters: debuggers let you explore state instead of reason about it. Reasoning is wrong often — you assume a variable is X when it’s actually Y. The debugger shows you the Y directly. Hours saved.

A weekend of practice

Sit down once, with a cup of coffee, and walk through your debugger’s features on a real bug. Set a breakpoint. Step through. Set a conditional breakpoint. Evaluate an expression mid-execution. Inspect the stack.

Most people never spend this two hours. It’s the highest-ROI two hours you’ll spend on tools this year.

Common pitfall — Print-debug loops

Add a print. Run. Add two more. Run. Add three more. Run. Remove them all. Put some back. Lose track of what you’ve added where.

Better — Breakpoint-driven debugging

One breakpoint. Inspect. Step. Iterate live. No clutter in the code. No forgotten console.log making it to production.

05 · Reading

Ramping Up on an Unfamiliar Codebase

You’ll join a new team, own a new service, or inherit a feature you’ve never seen. The ability to orient yourself fast in an unfamiliar codebase is a learned skill — a process, not a talent. The process beats the talented guesser every time.

Don’t start by reading all the code

The beginner’s instinct: open the repo, browse the folder tree, skim the files. After 30 minutes you feel lost. This is normal — random skimming is the worst way to learn a codebase.

The Ramp-Up Process

  1. Read the README first. If it’s useless, that’s your first proposal later.
  2. Run it. Get it building, booting, passing tests, serving requests — before you read any code. This teaches you the infra, the config, the setup.
  3. Pick one user-facing flow. Signup, search, checkout — whatever’s central. Trace it end-to-end from entry point to DB and back. This is your narrow tunnel into the architecture.
  4. Draw the flow on paper or a whiteboard. Three boxes, arrows. You’re building the mental model.
  5. Extend the tunnel to the adjacent flow. Then the next. Over a week, the tunnels merge into a map.
  6. Pick small tickets. Actual bugs or features in the area you’ve tunnelled. You learn fastest by doing, not by reading.

Scenario: ramping up on a service

🛠

You inherit an order service. 80 files. You’ve never seen it.

Don’t: start reading files alphabetically.

Do:

  • Get it running locally, even if setup takes a day.
  • Send a request that creates an order. Watch the logs.
  • Trace: controller → service → repository → DB. Five files, ten minutes.
  • Now send a request that fetches an order. Same tunnel.
  • Now send a request that cancels one. Now you know cancel paths.
  • After a week of narrow tunnels, you understand the service’s shape — and you read at most 15 of the 80 files. The rest you’ll learn as needed.
Common pitfall — Broad reading, no anchor

You open 20 files, nothing sticks because nothing’s connected yet. You feel overwhelmed and shut the laptop.

Better — One tunnel, end-to-end

Pick one flow. Trace it through every layer. Connections build a model. A single tunnel teaches you more than a week of browsing.

Common pitfall — Reading code you can’t run

You study code statically. Your mental model is wrong in subtle ways that only running it would reveal.

Better — Get it running first

Even if setup is painful. The act of making it boot forces you to encounter the config, the dependencies, the wiring — things you’d otherwise ignore.

The “ask a tourist” move

Find a teammate who’s been on the team one-to-two years. Not a founder, not a newcomer. Ask for 30 minutes of a walkthrough: “show me how X actually works.” Two years is the sweet spot — they know the code and they still remember what was confusing. They’ll cut your ramp-up in half.

06 · Prod

Production Investigation Basics

Sooner or later you’ll be asked to investigate something in production. Your instinct might be to panic. Don’t — it’s the same scientific method, with a handful of extra tools: logs, metrics, traces, and deploy history.

The four places to look

The Prod Investigation Sources

  1. Logs. Search for the error, the user ID, the request ID. Read a window — 30 seconds before and after the incident.
  2. Metrics / dashboards. Did error rate spike? Latency? CPU? Memory? Match the spike time to any known event.
  3. Deploy history. “Did anything deploy just before the incident started?” is the single most valuable question in production debugging.
  4. Traces (if you have them). A single request’s full journey through services. Incredibly powerful when present.
🎯

The deploy-history question: a disproportionate share of production incidents correlate with a recent deploy. “What changed?” is the first question, not the last. If something deployed in the 15 minutes before the incident started, that’s your prime suspect.

The “correlation is not causation” trap

You’ll find a metric that spiked. It might be the cause — or the effect. A spike in 500 responses could cause a spike in retries (one driving the other). Know the direction of causation before you act. Usually: the first thing that moved is the cause; everything that moved after is downstream.

Common pitfall — Searching logs with no time boundary

“Grep for errors.” You get 10,000 hits across three days.

Better — Narrow by time and context

“Between 14:30 and 14:45, errors for user X, in service Y.” Now you get 20 hits you can read.

Common pitfall — Assuming your service is the cause

Your service is erroring. Must be your bug. But actually the database is under pressure, or an upstream is flaking.

Better — Check upstream and downstream

What do my dependencies look like right now? Is the DB fine? Is the upstream service healthy? Often the bug isn’t in the code that looks broken — it’s in what that code depends on.

The runbook shortcut

If your service has a runbook, read it first. It’s the accumulated wisdom of every previous incident. If your service doesn’t have a runbook, write the first entry after you resolve this one. “Symptom: X. Likely cause: Y. Checks: Z.” Three lines save the next person an hour.

Key Insight

Production calm comes from practice, not talent. The first few prod investigations feel like panic; the tenth feels like a routine. Run through them with the method, contribute to the runbook, and over a year you become the person others want on the call.

Further reading

  • Debugging — David Agans. Short, canonical, nine rules that cover 95% of debugging problems.
  • Site Reliability Engineering (Google SRE Book, free online). Specifically the chapters on monitoring, incidents, and postmortems.
  • Julia Evans’ zines — Help! I have a bug! and How Containers Work are especially good at this layer.
  • Brendan Gregg’s blog (brendangregg.com) — the reference for performance debugging on Linux.

One thing to try this week

  • On your next bug, write down the hypothesis before you test it.
  • Spend an hour in your debugger on a real bug; use a conditional breakpoint.
  • When you join a new area of code, pick one user flow and trace it end-to-end.
  • When you hit a stack trace, read every frame at least once before opening a file.
  • Ask “what changed recently?” at the start of your next prod investigation.
