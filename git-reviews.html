Git, MRs & Code Review

Git hygiene. Small, reviewable MRs. Self-review before you ask for review. Reviewing others as part of the job. Receiving review without defence. The habits that make team code reliable — not just yours.

01 · Git

Git Hygiene

Git is your time machine. Good git habits make that time machine useful; bad ones turn it into a confused log that nobody wants to read. At this stage, you should have a handful of git habits you hold without thinking.

The core habits

Git Basics You Actually Use

  1. Small, meaningful commits. One logical change per commit. Not “WIP” x 12.
  2. Commit messages that future-you can read. A subject line under 70 chars, describing what and why.
  3. Branches with meaningful names. sujit/refactor-order-service beats temp-fix-2.
  4. Rebase before merge (if your team does rebase) — history stays linear and readable.
  5. Don’t force-push shared branches. Your own feature branch? Fine. main or a branch others are working on? Never.
  6. git status before anything destructive. Know what you have before you blow it away.

Writing a commit message

A good commit message has a subject line and, if needed, a body. The subject line is a summary in imperative mood (“Fix X”, “Add Y” — not “Fixed” or “Fixing”). The body explains the why if it’s not obvious from the diff.

Common pitfall — Useless messages

“fix”, “update”, “wip”, “.”, “address review comments”. Tells the reader nothing.

Better — What and why

“Parallelise payment-service calls to cut checkout latency”. Subject tells you what. Body (optional) tells you why it was slow and what approach you took.

Common pitfall — The “fixup fixup fixup” trail

Every time you fix a review comment, it’s a new commit. Final MR has 14 micro-commits. Nobody wants to bisect this.

Better — Squash or fixup before merge

Either squash at merge time, or use git commit --fixup and git rebase -i --autosquash to fold review fixes into the original commits. Clean final history.

Common pitfall — Huge commits that do three unrelated things

One commit refactors the service, fixes a typo, and adds a new feature. When something breaks, git bisect is useless.

Better — One logical change per commit

Refactor in one commit. Fix typo in another. Feature in a third. History becomes a ladder of small, reversible steps.

⚠

The force-push rule: if a branch is yours alone, --force-with-lease is fine. If anyone else might have pulled it, don’t. Force-pushing main is almost always a mistake; many teams block it entirely.

Further reading

  • Chris Beams, “How to Write a Git Commit Message” — the classic blog post on commit hygiene.
  • Pro Git (free online) — the canonical reference. Chapters 2–3 cover 90% of daily use.
  • Linus Torvalds’ talk on git — YouTube, 2007. Old but illuminates why git is shaped the way it is.
02 · Size

Small, Reviewable MRs

The single biggest thing you can do to make your code review experience better is to submit smaller MRs. Small MRs get reviewed faster, reviewed better, and merged sooner. Large MRs rot in the queue, get rubber-stamped, and introduce bugs everyone missed.

Why size matters so much

Reviewer attention is finite. A 50-line MR gets close scrutiny. A 500-line MR gets skimmed. A 2000-line MR gets an “LGTM” and a hope. The actual quality of the review is roughly inverse to size. This is measured, consistently, across teams.

Small also means: easier to revert, easier to rebase, easier to understand in git history. Large MRs are an anti-pattern that costs the team even after they merge.

Key Insight

Aim for MRs that a reviewer can deeply understand in 15 minutes. That’s usually 200–400 lines of real change (not generated files, not pure deletions). Anything over 600 is a candidate for splitting.

How to keep MRs small

The Small-MR Playbook

  1. Refactors before features. If you need to refactor to make the feature cleaner, do the refactor in a separate MR first. Merge. Then the feature MR.
  2. One concern per MR. Refactor, feature, or bug fix — not two of them, and never three.
  3. Stack branches. Feature branch A depends on the pending refactor branch. Review and merge A after the refactor.
  4. Use feature flags. Ship half-built code behind a flag so you can land it in pieces instead of one giant drop.
  5. Scope aggressively. The things you’d “like to also fix” — open follow-up tickets, don’t include them.

The “but it has to be big” case

Sometimes it really does. A migration. A framework upgrade. A ground-up rewrite. In these cases: warn the reviewer ahead of time, write a description that’s roughly longer than the diff summary, suggest the order to read files, and point out the three places that actually deserve attention. Make it as small as it can be for the task, not artificially small.

Common pitfall — “While I was there”

You open a file to fix a bug, and end up also renaming three things, extracting a function, and adding a log. Your 20-line bug fix is now a 400-line MR. The reviewer has to disentangle.

Better — Two MRs

MR 1: bug fix, 20 lines, merges fast. MR 2: the cleanup you noticed, separately. Reviewer can focus on each.

Common pitfall — “Let me finish the whole feature first”

You work on a branch for two weeks. You open a 2000-line MR. Reviewers look at it, then at their calendars.

Better — Merge early, merge often

First piece merges on day 2. Second piece on day 5. Third piece on day 8. Each is reviewable. You’re never far from main.

03 · Self-Review

Self-Review First

Before you click “request review”, read your own MR top to bottom in the MR view. Every time. This one habit eliminates maybe half the comments your reviewer would otherwise make.

Why self-review works

The MR view shows you your code the way a reviewer will see it — without the context of your local editor, your recent changes, your understanding of what you meant. Things you didn’t notice while writing jump out: debug console.log you forgot to remove, commented-out blocks, a variable renamed inconsistently, a confusing name, a missing test.

The Self-Review Pass

  1. Read the full diff in the MR view. Not in your IDE.
  2. File by file, line by line. Slowly.
  3. For each change, ask: does this change make sense on its own? Without the context of what I was doing.
  4. Look for leftovers. Debug prints, commented-out code, dead variables, TODO comments left behind.
  5. Check the description. Does it accurately say what the MR does? Does it explain why?
  6. Leave comments on your own MR where a reviewer might ask. “I left this in because…” or “I’m unsure about…” — preempt the questions.
💡

Why this matters: the reviewer’s time is the team’s scarcest resource. Every nit you catch yourself is a nit they don’t have to surface — which saves a round-trip and keeps the review focused on substance. Self-review is the cheapest way to respect reviewer time.

Scenario: the “oh” moment

You open your MR. First thing you see: a file you didn’t mean to commit — your editor config. Two seconds in, you’ve caught a thing that would have cost the reviewer a minute to flag and you a minute to unstage. Multiply by everything you catch. The ROI is obvious the first time you try it.

Common pitfall — Pushing and requesting immediately

You finish coding, commit, push, hit “request review.” The reviewer finds two debug prints and a broken import.

Better — Read it yourself first

Even 5 minutes. You’ll remove the debug prints, notice the broken import, and improve one variable name. The review that follows is about real substance.

04 · Giving Review

Reviewing Others’ Code

Reviewing code is part of your job, not a favour you do. It’s how the team maintains quality, how knowledge spreads, and how you learn by reading code you didn’t write. Treat it as first-class work, not an interruption to your “real” work.

What you’re reviewing for

What to Look For

  1. Correctness. Does it do what it claims? Are the edge cases handled?
  2. Clarity. Will someone else understand this in six months?
  3. Safety. Tests? Error handling? Security concerns (input validation, auth, secrets)?
  4. Design. Is the shape right, or is it building on a questionable abstraction?
  5. Consistency. Does it fit with how this codebase works, or does it introduce a third way?

Whitespace and formatting are not on this list. That’s what the linter is for.

How to phrase comments

Reviews are not performance reviews. Phrase comments as curious, collaborative observations — not demands. The code is the author’s, not yours.

Command tone

“Change this to X.” “This is wrong.” “Don’t do that.”

Curious tone

“What do you think about X instead?” “I wonder if this handles empty input — worth checking?” “Could we also consider Y?”

Blocking every nit

10 comments, all style, all “must change.” Author gets 10 round-trips for cosmetic issues.

Label severity

Prefix: nit: for style/taste (optional), q: for questions, blocking: for things that must change before merge. Author knows what to act on immediately.

Reviewing only what’s wrong

Only negative comments. Author gets 14 “fix this” and zero “this is good.”

Note the good

“Nice extraction here.” “This test covers an edge case I wouldn’t have thought of.” Fifteen seconds. Makes reviews better feedback loops.

Review turnaround

A review that takes three days costs more than the reviewer realises. The author has context-switched. Re-loading that context on day three costs an hour. The team’s velocity is gated by review latency more than it is by coding speed.

🕑

A reasonable target: review MRs the same day, ideally within a few hours. If you can’t, say so — “I’ll look at this tomorrow morning” is better than silence, and the author can find another reviewer if they need it sooner.

When you don’t know enough to review

Sometimes an MR touches an area you don’t know well. That’s fine — you can still review parts. Comment on the parts you understand. Flag explicitly: “I reviewed X and Y; I didn’t feel confident about Z, maybe worth a second look from someone who knows that area.” Partial review with honesty beats a blanket LGTM.

05 · Receiving

Receiving Review Well

When feedback comes back on your MR, every instinct says “explain, defend, argue.” Most of that is wrong. Learning to receive review gracefully makes every reviewer want to review your MRs — and the quality of your code goes up.

The core moves

Receiving Review Gracefully

  1. Assume good intent. The reviewer is trying to help, even when phrasing lands harsh.
  2. Don’t argue in the thread unless necessary. Most disagreements are resolved faster with a 5-minute chat than 10 round-trips of comments.
  3. If you agree, just fix it. No need to justify the original choice.
  4. If you disagree, say why — briefly. “I chose X because Y. Happy to change if you still prefer Z.”
  5. When you resolve a comment, make sure the resolution is visible. Either reply “fixed” or rely on the tool’s resolve mechanism.
  6. Thank the reviewer. Their time is a gift. A brief “thanks, good catch” on a nice comment is fifteen seconds you won’t regret.

When you think the reviewer is wrong

Sometimes they are. That’s fine. But before you reply combatively, assume 30% chance they’re right and you’re missing something. Re-read the comment. Consider their suggestion. Then reply.

💬

If it’s genuinely a disagreement worth escalating: move to a chat. Async back-and-forth on a PR comment is the slowest medium for disagreement. A 3-minute call often resolves what would be ten comments.

Common pitfall — Arguing every comment

You reply to 12 of 14 comments with “I did it this way because…” Reviewer stops giving feedback.

Better — Fix the easy ones, discuss the real ones

10 of 14 are easy — just fix. The 4 that are real disagreements, discuss. Reviewer sees focused engagement, not resistance.

Common pitfall — Silent resentment

You fix everything but grumble about it afterwards. The reviewer doesn’t know. You do it again next time. Rinse, repeat.

Better — Raise it once, clearly

“Hey, I’ve noticed you push back a lot on the way I structure X — can we align on a convention so we’re not going back and forth each MR?” One conversation fixes ten MRs.

06 · Description

The MR Description That Makes Review Easy

A good MR description doubles the reviewer’s effectiveness. It answers the questions they were going to ask, shows them the order to read in, and gives them the context to make a judgement instead of a guess. Five minutes writing it saves fifteen minutes of review confusion.

The template

A Good MR Description

  1. What this does — one or two sentences. Link to the ticket.
  2. Why — the problem, the motivation. Especially important if the “what” alone wouldn’t be obvious.
  3. Approach — high-level shape. What you changed where, why this approach over alternatives you considered.
  4. How to test / verify — how the reviewer can check that it works, or what they should look for if they want to click through.
  5. Any open questions — things you’re unsure about, or specific points you want a reviewer’s eye on.
  6. Screenshots / videos — for anything visible. A 10-second screen recording is often worth 200 words.

Scenario: two MR descriptions, same change

🛠

Description A: “Fix order processing bug.”

Description B:

What: Fix race condition in OrderProcessor where two concurrent cancel requests could both succeed. Closes TICK-412.

Why: Prod logs showed 3 duplicate refunds last week; root cause is the check-then-act pattern in cancelOrder. The second request passed the “is not cancelled” check before the first request had committed.

Approach: Replaced the check-then-act with an atomic conditional update (UPDATE … WHERE status = 'ACTIVE'). If zero rows affected, the request is a no-op. Considered pessimistic locking, but the update pattern is simpler and doesn’t add contention elsewhere.

Testing: Added a concurrent-request test that reproduces the old behaviour (fails) and verifies the fix (passes). Also manually fired two parallel curl requests on staging.

Open question: should we audit-log the no-op? Left it out for now — happy to add if you think it’s worth it.

Description A requires the reviewer to figure out everything. Description B gives them the context to review in 10 minutes. Five minutes of writing, massive savings on the other side.

Common pitfall — Empty or one-line description

Reviewer opens a 300-line diff with no context. They have to reverse-engineer the intent. Review takes 40 minutes.

Better — The five-minute description

Tell the reviewer what, why, how, and what to look at. Review takes 15 minutes. They actually check the right things.

Common pitfall — No screenshots for visual changes

UI MR. Reviewer has to pull the branch and run it to see what it looks like. Friction. Half won’t bother.

Better — Before/after screenshots

Two images or a short GIF. Reviewer sees the change instantly and knows what’s been checked visually.

Key Insight

Your MR description is an investment in the reviewer’s time. A great description costs you five minutes and saves the reviewer fifteen. Over a year of MRs, that’s days of their time reclaimed. Team-wide, it’s how review-turnaround culture is built.

Further reading

  • Google’s Code Review Developer Guide (free, public) — the best industry-wide reference, covers both author and reviewer sides.
  • Michael Lynch, “How to Do Code Reviews Like a Human” (two-part blog post).
  • What to Look for in a Code Review — Trisha Gee’s short eBook (free from JetBrains).

One thing to try this week

  • Write a full MR description (what, why, how, testing) on your next PR.
  • Before asking for review on your next MR, read it top to bottom in the MR view.
  • Prefix your next review comments with nit:, q:, or blocking:.
  • Leave one specific positive comment on a peer’s MR this week.
  • Squash your MR’s fixup commits before merge — clean final history.
