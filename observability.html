Observability & Errors

Code that runs in production is a black box until you teach it to talk. Logs, metrics, traces, and error handling aren’t something you bolt on during incidents — they’re something you design into every commit. This chapter is about making your systems legible from the outside.

01 · Basics

The Three Pillars (And Why)

Observability is usually broken into three pillars: logs, metrics, and traces. They answer different questions. Your job is to know which tool answers which question and not to use one when another would be better.

PillarAnswersShape
LogsWhat happened in this specific event?Unbounded, detailed, text-heavy. Expensive at scale.
MetricsWhat’s the rate, shape, and trend?Numeric, aggregated, cheap. Cardinality-limited.
TracesWhat was the path of this request across services?Structured, per-request, expensive raw but sample-able.

Key Insight

Metrics tell you something is wrong. Logs tell you why it’s wrong for this case. Traces tell you where in the chain it went wrong. Use all three; don’t try to make any one do the others’ jobs.

02 · Logs

Logging That Actually Helps

Most logs are noise. The engineer who writes useful logs is the engineer whose 2am incident takes 20 minutes instead of 4 hours. Useful logs have a specific shape.

The Useful-Log Rules

  1. Use structured logging. Log lines should be parseable — JSON, key-value, whatever the platform expects. Text blobs don’t scale.
  2. Include context. Request ID, user ID (not PII), tenant, operation name, and the values that let you re-run the failure in your head.
  3. Log at boundaries, not everywhere. Entry to a service, exit to a downstream call, decisions that affect flow. Logging every line adds noise and cost.
  4. Use levels deliberately. ERROR for real failure, WARN for recoverable, INFO for normal flow, DEBUG for devs. Production runs at INFO; DEBUG is for local and short investigations.
  5. Correlate. Include a correlation ID that follows a request across services. Without it, debugging a distributed call chain is guesswork.
  6. Log the cause, not just the symptom. “DB connection refused” beats “save failed.” Log the original exception when you re-throw.

A scenario: the silent save

🔎

Users complain their updates sometimes don’t save. Your logs say: INFO: Save attempted. INFO: Save complete. For every request. Every one “succeeded.” No error logs. You can’t reproduce. You’re stuck.

What useful logs would have said: INFO: Save attempted. user_id=123, tenant=acme, op=update_profile, fields=[name, email]. WARN: Optimistic lock version mismatch — retrying. INFO: Save succeeded on retry. retry_count=1. Now you see the real story: retries are masking a race condition, and some attempts are exhausting retries without alerting.

Patterns & anti-patterns

Common pitfall — Log-and-swallow

catch (e) { log("oops"); }. The log is useless (no exception), the failure is invisible, and execution continues as if nothing happened.

Better — Log with the exception and re-throw (or handle explicitly)

If you swallow, you own the silence. Log the exception with context, re-throw it, or handle it with a documented decision. Never both log and silently continue.

Common pitfall — Logging giant objects

You log the whole request body “just in case.” Now your log storage is a full mirror of every request, your search is slow, and PII is everywhere.

Better — Log the fields that matter

The IDs, the decision branches, the error-relevant values. Not the password. Not the full payload.

Common pitfall — Everything at ERROR

Expected failures (user typed a bad password) log as ERROR. Real errors drown in noise. Your on-call mutes the alert.

Better — ERROR is for things we’d page on

User error = INFO or WARN. System error we can’t handle = ERROR. Keep ERROR rare and real — it’s a signal, not a volume control.

03 · Redaction

What Not to Log (And Why)

A log file is the most common accidental data leak. The moment data hits your logs, it hits your log index, your backup, possibly your SIEM, possibly a third-party log processor. If it’s sensitive, it’s now everywhere sensitive data shouldn’t be.

The Never-Log List

  • Passwords. Plain, hashed, encrypted — doesn’t matter. Not in logs.
  • Auth tokens, API keys, session cookies.
  • Credit card numbers, full or partial beyond last-4 when legally required.
  • Full PII in free text — addresses, phone numbers, government IDs.
  • Personal health data, financial data subject to regulation.
  • Request bodies and response bodies, unsanitised.
  • Full exception objects that might contain the above.
⚠

Build redaction into your logging framework, not into every call site. A helper that logs a user: logger.info("user action", user=user) should already know to log only id, tenant, role. Relying on every engineer to remember to sanitise is how leaks happen. Centralise the policy; every call site gets it for free.

A habit: log review as part of review

When you review a teammate’s MR, add this to your mental checklist: “What will this log, and is any of it sensitive?” Most leaks come from someone wiring a new logging call without realising what it prints. A 10-second check in review catches almost all of them.

04 · Metrics

Metrics: The Shape of the System

Metrics are where you see trends, thresholds, and saturation. They’re cheap, always-on, and the first place to look when something feels off. Good metrics answer specific questions; bad metrics are a dashboard graveyard nobody reads.

The four golden signals

Latency

How long did the thing take? Track p50, p95, p99 — averages hide the tail that hurts users.

Traffic

How many requests, transactions, events? You need to know baseline to know when abnormal.

Errors

What fraction failed? Error rate vs absolute count — both matter. A 0.1% rate at 1M requests is a lot.

Saturation

How full are the resources? CPU, memory, connections, queue depth. The leading indicator of the next incident.

Track these four for every service you own. Plot them on one dashboard. Look at them before you claim something is healthy.

Cardinality: the trap

💡

Don’t put user IDs, request IDs, or unique identifiers in metric labels. Metrics are pre-aggregated; each unique label combination is a time series. Add user_id as a label, create one series per user, watch your monitoring bill explode. Labels are for low-cardinality dimensions: endpoint, region, status code, tenant (if small).

Business metrics, not just technical ones

Technical metrics (latency, errors) tell you the system is up. Business metrics (orders per minute, signups per hour, successful checkouts) tell you it’s working. A system can have 0% error rate and a broken business outcome — a silent fallback, a wrong feature flag, a misconfigured routing rule. Track the things that matter to the product, not just to the servers.

05 · Tracing

Tracing: Following One Request

In a system with more than one service, tracing is what lets you follow a single request across hops. “API took 2 seconds” becomes “API took 2 seconds because the auth call took 1.8 and most of that was DNS.” Without traces, you’re guessing which link in the chain is slow.

The Tracing Habits

  1. Generate a trace ID at the entry point and propagate it through every downstream call — HTTP headers, queue messages, background jobs.
  2. Include it in your logs. One ID ties logs across services together. An investigator can follow one request end-to-end.
  3. Instrument the calls you actually care about: outbound HTTP, DB queries, cache lookups, queue publishes. Libraries usually auto-instrument the common ones.
  4. Sample sensibly. Full tracing is expensive. Sample representative traffic plus 100% of errors. That gives you the detail where it matters and low cost on the happy path.
🌱

Worth the one-time cost: setting up distributed tracing is a multi-day project on a team that doesn’t have it. The day after it’s working, a bug that took a week to find takes 10 minutes. The investment compounds — every incident after that is faster.

06 · Errors

Error Handling Philosophy

Most bugs in production aren’t in the code that runs when things go right — they’re in the code that runs when things go wrong. Error handling isn’t a chore; it’s half your job. Treat it as a design concern, not an afterthought.

The questions to ask for every error

The Error Triage

  1. Is this expected or unexpected? User typed a bad password: expected. Database connection refused: unexpected. They’re handled differently.
  2. Is this recoverable? Can we retry, fall back, or degrade gracefully? Or do we have to stop?
  3. Where is this best handled? At the point it happened, or at a higher boundary where there’s enough context to make a real decision?
  4. What does the caller need to know? An end user needs a friendly message; an internal service needs a structured error; a developer needs the full stack.
  5. What does the system need to know? Metric? Log? Alert? All three? None?

The three places to handle errors

At the source

When you have enough information to make the decision now. Retry a transient DB blip. Default a missing config. Handle and move on.

At a boundary

Request handlers, message consumers, scheduled jobs. The place where you can translate “something broke” into the right response — error to the user, failure metric, message for retry, whatever fits.

Nowhere (let it crash)

Sometimes the right thing is to fail, let the process die, and let your platform restart it. Erlang made this famous. For long-running pipelines it’s often the cleanest answer.

Key Insight

The worst error handling is the middle of the stack catching something it can’t really handle, logging a message, and returning a default. The error is invisible to the caller, the logs accumulate, the system appears to work. Either handle it with a real decision or don’t catch it — passing the problem up is better than hiding it.

07 · Fail Loud

Fail Loud vs Fail Soft

A failure that screams is a failure you can fix. A failure that whispers is a failure that sits in production for months, corrupting data, disappointing users, until someone notices. Default to loud — especially for engineers still building their instincts.

Common pitfall — The silent default

let x = parseInput(input) ?? 0;. When parsing fails, x is 0. Math proceeds. Reports look “fine.” Three months later someone notices totals are wrong.

Better — Default with intent, or throw

If 0 is a meaningful answer for bad input, fine — but decide on purpose and log the substitution. If it isn’t, throw. Invisible defaults are how data corrupts quietly.

Common pitfall — try/catch for flow control

You use exceptions to drive normal logic. Now “caught” and “never happened” look the same in the logs.

Better — Exceptions for exceptional

Expected results (“user not found”) are return values, optionals, or result types. Exceptions are for things you don’t know how to handle — loud by default.

Common pitfall — Empty catch

catch (Exception e) { /* ignore */ }. The worst line of code in the history of software.

Better — Every catch has intent

If you’re catching it, you have a reason — write a comment naming the reason, log it, and make a real decision. If you can’t articulate why you’re catching it, don’t.

When “soft” is right

There are cases where failing soft is correct: optional features, analytics, non-critical enrichment, UI degradation. The rule: soft failures must be explicit and visible. A metric counter, a log line at WARN, a degraded badge on the dashboard. Silent soft failure is still silent — and silent is the problem, not soft.

08 · Alerts

Alerts That Earn Their Ring

An alert that wakes someone up at 3am should be worth waking them up for. If it isn’t, the alert is the problem, not the engineer who slept through it. Alert quality is a design concern.

The Alert Quality Rules

  1. Alert on symptoms, not causes. “User-facing error rate > 5%” is a symptom. “CPU > 80%” is a cause that may or may not matter. Alert on the thing users experience.
  2. Every alert has a runbook. If an alert fires and there’s no runbook, write one. If you can’t write a runbook, the alert isn’t actionable — delete it.
  3. Every alert has an owner. “The team” is not an owner. A specific on-call or rotation.
  4. Alerts that fire routinely without action get tuned or deleted. Alert fatigue isn’t a personality flaw; it’s a design flaw.
  5. Page only for things a human must do something about now. Everything else is a ticket, a dashboard, or an email.

Dos & Don’ts

DoDon’t
Log structured, with correlation IDs and contextLog plain text blobs without context
Track the four golden signals for every serviceCollect metrics you never look at
Redact sensitive fields in a central logging helperRely on each call site to sanitise
Fail loud by default; make soft failure explicitSwallow exceptions with an empty catch
Alert on user-visible symptoms with runbooksAlert on every metric crossing a round number
Sample tracing smartly: representative + all errorsTrace 100% and pay for it forever, or 0% and debug blind
Include a trace ID in every log lineStore logs and traces in disconnected tools
🔮

The compounding payoff: good observability is invisible when everything works and invaluable when anything breaks. The engineer who built it in early moves through incidents in minutes while others are still asking “wait, why didn’t this log anything?” An hour spent on observability at the start pays back a hundredfold the first time production misbehaves.

Further reading

  • Site Reliability Engineering (Google SRE Book) — the chapters on monitoring and SLOs define the modern vocabulary. Free online.
  • Observability Engineering (Majors, Fong-Jones, Miranda) — the current reference. Strong on why three pillars aren’t enough and what replaces them.
  • Charity Majors’ writing (charity.wtf) — opinionated, sharp, about ten years ahead of most teams.
  • Distributed Systems Observability (Cindy Sridharan) — short, free, excellent introduction.
  • OpenTelemetry docs (opentelemetry.io) — the practical instrumentation reference.
  • Baron Schwartz’s blog — deep, practical writing on metrics and performance.

One thing to try this week

  • Open the last log line your code wrote. Does it include a correlation ID? If not, add one.
  • Find one catch in your code that swallows silently. Fix it — log with context and re-throw, or make the handling explicit.
  • Check one production metric dashboard. If it’s not tracking p95/p99 latency and error rate, add them.
  • Pick one alert that fires without anyone acting on it. Tune it, route it to a non-page channel, or delete it.
  • Grep your logs for obvious PII (emails, full names, tokens). Add filtering at the logging layer for anything you find.
