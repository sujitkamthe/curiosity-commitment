Security Hygiene

Security isn’t a separate team’s problem. When the dev team owns everything, security is your problem — baked into every commit, not bolted on before release. This chapter is about the everyday habits that keep your code, your users, and your org safe.

01 · Why

Why This Chapter Exists

Many teams have a security team that runs scans, approves releases, and flags issues. Yours doesn’t. That means every commit is also a security decision — made by you, usually without noticing. Getting the small habits right is how you avoid the expensive accidents.

Key Insight

Security bugs are usually boring: a secret in a repo, a missing permission check, a SQL query built from a string. They don’t require creativity to prevent — they require habits. Build the habits now and you’ll rarely meet the exciting version.

The default stance

  • Assume input is hostile — from users, from other services, from yesterday’s migration script.
  • Assume least privilege — what’s the minimum permission this code needs to work? Start there and only expand with reason.
  • Assume breach is possible — if an attacker got in today, what would they see? Design the blast radius before you need it.
  • Assume you’ll make mistakes — put in layered defences so a single mistake isn’t catastrophic.
02 · Secrets

Secrets & Credentials

The most common security accident is still the most preventable: a secret in a git repo. Keys, passwords, tokens, connection strings — once they’re in history, treat them as public.

The rules

The Secrets Rules

  1. No secrets in code, config files in the repo, or commit messages. Ever. Not even a test key.
  2. Use a secrets manager — Vault, AWS/GCP Secret Manager, Kubernetes Secrets with care, a dedicated tool. Your app reads at runtime.
  3. Local dev uses a .env that’s gitignored, never committed. Provide a .env.example with placeholder values.
  4. Rotate on suspicion. If a secret might have been exposed, rotate it. Takes an hour; saves a breach.
  5. Rotate on schedule for sensitive ones. Auth tokens, database credentials, cloud keys — on a cadence, not “when we remember.”
  6. Pre-commit scan. Run a tool (gitleaks, trufflehog) locally or in CI that blocks commits with secrets. Make the default path safe.

The “oh no” checklist

You just realised you committed a secret.

⚠
  1. Rotate the secret immediately. Not after you clean the repo — first. Removing it from git does not remove it from every clone, mirror, backup, and log.
  2. Tell the team. Slack, security channel, whatever you have. Fast transparency is the right move.
  3. Invalidate anything the secret could have touched. Sessions, cached tokens, downstream dependencies that use it.
  4. Clean the repo history only after rotation. git filter-repo or equivalent. Force-push if private; harder if public.
  5. Add a pre-commit scan so this shape of mistake can’t repeat.
Common pitfall — “It’s only a test secret”

A key that’s “only for staging.” Committed. Leaked. Turns out staging touches real customer data, or shares a DB with prod, or the key is valid for internal APIs.

Better — All secrets are real secrets

Test keys, dev keys, “just for now” keys — they all go in the secrets manager. If something’s not a secret, it’s a config value, and that’s fine to commit.

03 · Input

Never Trust Input

Every injection bug has the same shape: user data flowed somewhere it wasn’t supposed to flow — a database query, a shell command, an HTML output, a log line that gets parsed. The fix is always the same: treat the boundary as a boundary.

The Input Rules

  1. Parameterise every query. Use prepared statements / bound parameters. Never build SQL with string concatenation. Never.
  2. Escape at the output layer, not the input layer. HTML-escape when rendering HTML, shell-escape when calling shell. Input validation is in addition, not instead.
  3. Validate shape and range. If you expect an integer 1–100, reject anything else at the boundary. Don’t trust what’s declared; verify.
  4. Whitelist where you can. Enum-safe choices beat free-text choices.
  5. Reject early, in one place. Validate at the entry point (controller, handler). Inner code can then assume valid input.
  6. Avoid eval, exec, and dynamic code paths on user data. If you need it, treat it as a red flag — lock it down or find another way.

The classic shapes to recognise

SQL Injection

"SELECT * FROM users WHERE id = " + input. Fix: prepared statements. Every time.

XSS (Cross-Site Scripting)

User-supplied text rendered into HTML without escaping. Fix: render through a templating engine that auto-escapes, or escape explicitly.

Command Injection

exec("convert " + filename). Fix: pass as arguments, not strings. Use APIs that accept arrays, not shell strings.

Path Traversal

User provides “filename”, you open ./uploads/" + filename. They pass ../../etc/passwd. Fix: resolve, normalise, check the result is still inside the allowed directory.

Unsafe Deserialization

Parsing untrusted YAML/pickled/serialised blobs that can execute code. Fix: use safe parsers; don’t deserialise untrusted data into live objects.

SSRF (Server-Side Request Forgery)

User provides a URL, your server fetches it. They provide http://169.254.169.254 (cloud metadata). Fix: whitelist destinations; block internal addresses.

💡

A habit that catches 90% of these: when you write code that mixes user input with SQL, shell, HTML, URLs, or filesystem paths — stop. Ask: “what’s my escape or parameter mechanism here?” If you can’t name it, you probably don’t have one.

04 · Authn / Authz

Authn vs Authz

Two different problems, often confused. Authentication is “who are you?” — proven by password, token, or key. Authorisation is “what are you allowed to do?” — enforced at every operation. Most bugs are on the second one.

Authentication habits

  • Never implement your own auth primitives. Use the library your platform provides. Password hashing, token signing, session management have one-way trapdoors you don’t want to discover yourself.
  • Use strong, salted, slow password hashing — bcrypt, scrypt, or argon2. Not MD5, not SHA-256. Not “SHA-256 twice.”
  • Assume tokens will leak. Short-lived tokens, rotation, the ability to revoke.
  • Protect login from brute-force and credential-stuffing. Rate-limit, monitor, alert on unusual login patterns.

Authorisation habits — the one that bites

The Authz Checklist

  1. Every endpoint has an explicit permission check. Not “authentication is required” — “this specific user can do this specific thing to this specific resource.”
  2. Check ownership on every resource access. If a URL includes a resource ID, verify the logged-in user is allowed to touch that resource. Don’t assume “they wouldn’t have the ID.”
  3. Default deny. A new endpoint without explicit permission setup should refuse access — not allow it.
  4. Separate admin paths from user paths. Admin operations use different endpoints, different middleware, different audit logging.
  5. Don’t rely on UI hiding. The UI hiding a button doesn’t stop anyone from calling the endpoint directly.
Common pitfall — IDOR (Insecure Direct Object Reference)

You fetch /invoices/42 and return it. You forgot to check the logged-in user owns invoice 42. User changes the number in their URL bar; now they can see anyone’s invoice.

Better — Ownership check on every resource fetch

invoice = findByIdAndOwner(42, currentUser). If the tuple doesn’t match, return 404 (not 403 — don’t leak whether the ID exists).

Common pitfall — Permission check in the UI only

The admin button is only shown to admins. Someone inspects the network call and replays it without being an admin. The endpoint didn’t check; it trusted the UI.

Better — Permission check on every endpoint

UI hiding is UX. Authorisation is a server-side concern. Both belong; they solve different problems.

05 · Dependencies

Dependencies Are Code You Didn’t Write

Most of the code in your service was written by strangers. A single npm install pulls in hundreds of transitive packages. Each one is a trust decision you’re making without reading. Small habits make that manageable.

The Dependency Hygiene Loop

  1. Commit the lockfile. package-lock.json, Cargo.lock, pom.xml, go.sum. Reproducible builds are non-negotiable.
  2. Automate vulnerability alerts. Dependabot, Renovate, Snyk, npm audit, or equivalent. A new CVE against your dependencies should create a ticket automatically.
  3. Patch on a rhythm. Security updates weekly or monthly — not “when we remember.” Small, frequent updates beat one big scary one every year.
  4. Pin direct dependencies; let transitive ones float within the lockfile. You control what you declare; the lockfile locks the tree you got.
  5. Be suspicious of obscure packages. Tiny download counts, recent creation, maintainer you can’t verify, names close to popular packages (“typo-squatting”). Don’t install without a reason.
  6. Read release notes on major updates. A minor version bump is probably fine; a major one might be changing security defaults. Ten minutes of reading saves real pain.
💡

The “left-pad” principle: every dependency is a commitment. The more you take, the more you trust. When you’re about to add one to do something trivial, ask: can I write this in 10 lines? Sometimes yes. Sometimes the library is the right call because it’s handling edge cases you’d get wrong. Decide on purpose, not by reflex.

06 · Data

Handling Sensitive Data

Most of the data your service touches — emails, names, addresses, identifiers, payment details, health or financial info — is someone’s. The rules for handling it are not optional: they’re contracts, often legal ones. Know the basics.

The Data Handling Rules

  1. Minimise. Don’t collect data you don’t need. Don’t store data you don’t need. The safest data is data you don’t have.
  2. Encrypt in transit. TLS everywhere, including internal calls. HTTP inside the cluster is not “safe” — treat it as the public internet.
  3. Encrypt at rest for sensitive columns — PII, payment, credentials. Most databases support this; enable it.
  4. Mask in logs. Never log full credit card numbers, full tokens, passwords, health details, or government IDs. See Observability — What Not to Log.
  5. Separate what’s needed for ops from what isn’t. Engineers need stats and counters in production, rarely raw user data. Build your debug tools around that distinction.
  6. Know your regulations. GDPR, HIPAA, PCI, or local equivalents — not memorised end to end, but know what applies to your product and where to look when it does.

Key Insight

The single most common data leak is not a clever attack — it’s a log file or an analytics event. The PII went there because someone logged the whole request body while debugging. Ten lines of filtering, added on day one, prevents the whole class.

07 · OWASP

The OWASP Top 10 — Short Form

The OWASP Top 10 is the industry’s list of the most common web vulnerabilities. You don’t need to memorise the whole document — you need to recognise the shapes. Here they are, short.

Broken Access Control

The IDOR-style bugs in section 4. Fix: permission check everywhere, default deny.

Cryptographic Failures

Weak hashing, no TLS, plaintext storage. Fix: use the right primitives, encrypt by default.

Injection

SQL, command, LDAP, NoSQL. Fix: parameters, not strings.

Insecure Design

The flaw is in the architecture, not the code. Fix: threat-model before you build.

Security Misconfiguration

Default passwords, debug mode in prod, overly permissive CORS. Fix: known-good templates, minimum config, audit.

Vulnerable Components

Old dependencies with known CVEs. Fix: the dependency loop in section 5.

Identification & Authentication Failures

Weak session management, predictable tokens, no MFA. Fix: use the library, don’t roll your own.

Software & Data Integrity Failures

Unverified update streams, unsigned artifacts, insecure deserialisation. Fix: verify what you install and what you parse.

Logging & Monitoring Failures

You’d have caught the breach if you’d been looking. Fix: see Observability.

SSRF

Server fetching URLs it shouldn’t. Fix: whitelist, block internal addresses.

🏆

A useful annual habit: once a year, skim the current OWASP Top 10 (it updates every few years). For each item, ask: “which of our services is most exposed?” Two hours. It’ll catch something.

08 · Threat-Model

Threat Modelling for Your Change

Threat modelling sounds heavyweight. At the scale of a single MR or design doc, it’s a 10-minute conversation that catches most of the mistakes. The question is: “what could go wrong here if someone tried?”

The Five-Minute Threat Model (STRIDE-lite)

  1. Spoofing — could someone pretend to be another user? Do we verify identity at every step?
  2. Tampering — could someone modify data in flight or at rest? Are we validating and signing where it matters?
  3. Repudiation — could a user deny an action they took? Do we have the audit trail to prove it?
  4. Information disclosure — what sensitive data does this touch? Where could it leak (logs, error messages, responses)?
  5. Denial of service — could someone make this expensive with a cheap request? Rate limits, quotas, bounded work?
  6. Elevation of privilege — could a user become an admin? Can input control control flow it shouldn’t?

When to do it

Common pitfall — Saving it for “security review”

You build the whole feature, it’s ready to ship, and now you think about threats. The fixes are expensive now. Some of them mean architectural changes you can’t make.

Better — Threat-model at design time

The five-minute model applies when you’re still drawing boxes and arrows. The fixes are cheap. Often they reshape the design; that’s when reshaping is free.

Dos & Don’ts

DoDon’t
Run a secrets scan in pre-commit and CIRely on memory to not commit secrets
Parameterise every query at the library levelEscape inputs by hand and hope
Check ownership on every resource accessTrust the URL or the UI to enforce access
Automate dependency scans and review weeklyUpdate packages once a year in a panic
Minimise the data you collect and retainStore everything “in case we need it”
Use the platform’s auth libraryImplement your own crypto, hashing, or token logic
Threat-model at design time, not review timeBolt security on at the end

Further reading

  • OWASP Top 10 (owasp.org) — free. Refresh your memory annually.
  • Web Security for Developers (Malcolm McDonald) — short, accessible, covers the basics at the right depth.
  • The Tangled Web (Michal Zalewski) — deeper, excellent on browser security.
  • Threat Modeling: Designing for Security (Adam Shostack) — the canonical text if you want depth.
  • Cryptography Engineering (Ferguson, Schneier, Kohno) — when you want to know why “don’t roll your own” is the rule.
  • tldrsec (tldrsec.com) — accessible weekly newsletter summarising what’s happening in security.

One thing to try this week

  • Run a secret-scanner (gitleaks, trufflehog) against your repo. Rotate anything it finds.
  • Pick one endpoint you own and verify every path has an explicit ownership check. Add one where missing.
  • Look at your latest log output. Find one instance of PII or a token leaking and filter it.
  • Run your dependency audit (npm audit, pip-audit, cargo audit, etc.). Fix one real vulnerability.
  • On your next design doc, add a five-minute threat-model section. Even a paragraph is more than zero.
