Shipping Safely

Every deploy is a risk — unless you deploy in a way that makes rollback cheap, exposure gradual, and mistakes recoverable. Feature flags, backwards-compatible migrations, and progressive rollout aren’t senior-engineer tricks. They’re how anyone who owns their service ships without fear.

01 · Premise

Why This Matters

Confident deploys aren’t a personality trait. They’re the outcome of a deploy process that makes bad outcomes cheap. The engineer who ships at 4pm on a Friday without anxiety is usually not braver — they’ve just built the safety nets.

Key Insight

Every line of code is a bet that you got it right. Shipping safely is about making sure the bet is reversible. If reverting is cheap and fast, you can take more bets. If reverting is expensive, every deploy becomes a ceremony and velocity collapses.

The question that drives this whole chapter

For every change you’re about to ship: “If this turns out to be wrong, how do I undo it in five minutes?” If the answer is “I can’t,” you need to change the shape of the change — not the confidence level.

02 · Small

Small, Frequent Deploys

The most underrated safety technique is deploying more often with less in each deploy. Research (see Accelerate) is unambiguous: teams that deploy often have fewer failures and recover faster. Small batches make every other technique on this page easier.

The Small-Deploy Moves

  1. One logical change per deploy. Not “the Friday release with seven features.” That’s how you lose an hour of incident debugging trying to figure out which one broke it.
  2. Merge small, deploy small. Your MR shape directly shapes your deploy shape. See Git, MRs & Review.
  3. Automate the path. If a deploy is “scary” because it takes 45 minutes and six manual steps, the answer is automation, not ceremony.
  4. No “save it for the release.” If it’s ready and the risk is understood, ship it. Batching just increases blast radius.
💡

The counter-intuitive part: more frequent deploys feel riskier but are usually safer. Each deploy has less in it, so each failure has a narrower cause. The team builds the muscle of shipping. Rollback happens so rarely that when it’s needed, nobody remembers how — but if you deploy five times a day, rollback is routine.

03 · Flags

Feature Flags

A feature flag lets you deploy code without enabling it. It decouples release (the code is live) from activation (the users see it). This is the single most powerful safety tool in modern engineering, and it’s underused by teams who think of flags as “for A/B tests.”

What flags unlock

Deploy early, activate later

Ship the code in a deploy that’s small and reversible. Turn it on separately when you’re ready.

Kill switch

New behaviour gated on a flag can be turned off in seconds without a deploy. A release that took hours to build can be reverted in one click.

Progressive rollout

Enable for 1% of users, then 10%, then 50%, then 100%. Monitor at each step. Catch issues at 1% rather than at 100%.

Targeted release

Internal users first, then a specific tenant, then broader. Dogfood before exposing.

Merge without releasing

Long-lived branches are painful. Merge the half-built feature to main behind a flag; keep iterating without branch drift.

Reversible migrations

Old and new behaviour coexist behind a flag during migration; flip back instantly if the new one misbehaves.

The flag lifecycle

The Full Flag Lifecycle

  1. Create — default off. Code reads the flag from a config or service.
  2. Deploy code — disabled; safe because inactive.
  3. Enable for internal users — dogfood. Collect feedback, fix obvious issues.
  4. Roll out progressively — 1%, 10%, 50%, 100%. Monitor metrics at each step.
  5. Monitor at 100% for a stable window. If stable, proceed to cleanup.
  6. Remove the flag. Delete the old code path, delete the flag config. A flag that stays forever is debt.
⚠

The “flag graveyard” problem: teams that add flags faster than they clean them up end up with hundreds of stale flags, none of which anyone dares touch. Own the cleanup. When you add a flag, put a follow-up ticket in the backlog to remove it once the feature is stable. Six months later the code is cleaner, not messier.

Patterns & anti-patterns

Common pitfall — Flag only in the happy path

The new code path is gated; the new error handler isn’t. When something goes wrong, the system hits the new error code anyway.

Better — Flag the entire code path

Entry to exit, including error paths, logs, metrics, and side effects. The flag should cleanly switch between two complete worlds.

Common pitfall — Flags as permanent config

A “feature flag” never gets removed. Five years later it’s load-bearing config that nobody remembers creating.

Better — Separate flags from config

Feature flags are short-lived. Config is long-lived. Use different tools, different naming, different lifecycles. Don’t let one become the other.

04 · Migrations

Backwards-Compatible Migrations

Database changes are the scariest kind of deploy because they’re often hardest to reverse. The good news: almost every change can be decomposed into steps that are each safe, reversible, and deployable independently. Do that decomposition, and migrations stop being scary.

The expand / migrate / contract pattern

Three-Step Schema Change

  1. Expand. Add the new column / table / index alongside the old one. No destructive change. Old code still works; new code can start writing to both.
  2. Migrate. Backfill data, dual-write, and shift reads gradually to the new shape. Verify. Take your time here — this is where bugs surface.
  3. Contract. Once reads and writes are fully on the new shape, drop the old column / table / code. Only then.
🌱

Why this works: at every step, you can roll back to the previous state. The old world and the new world coexist during the transition. If you discover a problem at step 2, you turn off dual-writing and go back to the old world — no data lost. Compare to a single-step “rename column and update all code at once” deploy — which is safe when it works and catastrophic when it doesn’t.

Specific migration hazards

Adding a NOT NULL column

Add the column nullable first, backfill, then add the constraint. “Nullable + default” is usually fine on small tables but not on huge ones where the rewrite is long.

Renaming a column

Don’t. Add a new column, dual-write, migrate readers, drop the old one. Renames look atomic and aren’t.

Changing a type

Add a new column with the new type. Backfill. Move readers. Drop the old.

Adding an index on a large table

Use the concurrent or online variant. Plain CREATE INDEX can lock the table for minutes.

Removing a column

Remove readers first, deploy. Confirm no writers reference it. Then drop in a second deploy. Never in the same deploy as the code change.

Large data backfills

Batch. Monitor. Respect the database’s load. A single UPDATE of 100 million rows will lock the table. Do it in chunks.

Key Insight

Every scary migration can be decomposed into a sequence of safe ones. If your change “can’t be done safely,” it usually means you haven’t finished decomposing it. Take another half-hour to break it down; the deploy will be boring instead of terrifying.

05 · Rollout

Progressive Rollout

Even when a change is behind a flag, flipping it to 100% at once is more exposure than you need. Roll out gradually, watch the metrics, and give problems a chance to surface while the blast radius is still small.

A Reasonable Rollout Schedule

  1. Internal users / your own account — minutes. Immediate sanity check.
  2. 1% of real traffic — hours. Enough volume to see error rates and tail latency.
  3. 10% — a day. Load and scale effects start to appear.
  4. 50% — a day or two. Now you’re running in proper production conditions.
  5. 100% — monitor for at least a day before declaring it stable.

Adjust to the risk profile. A UI copy change doesn’t need a week of rollout. A change to the payment processor does — and maybe at even finer granularity.

What to watch at each step

  • Error rate for the specific code path, not just overall service error rate.
  • Latency, p50 / p95 / p99, broken out by the affected endpoint.
  • Business metric tied to the feature — conversion, successful transactions, whatever this change is supposed to affect.
  • Error composition — new error types showing up in your logs?
  • Downstream calls — is the new path hitting dependencies differently? Quota issues?
💡

Before you advance to the next percentage, name the thing you’re looking for and then check it. “No alerts” is the weakest possible check — alerts are for problems you already knew about. Look at the specific new metrics for the feature you’re shipping.

06 · Rollback

Rollback as a First-Class Plan

Every deploy plan includes a rollback plan — written before the deploy, not improvised during it. If you can’t describe exactly how to undo the change, you’re not ready to do it.

The Rollback Plan

  1. What’s the trigger? What metric or signal says “roll back”? Decide in advance; don’t negotiate during the incident.
  2. What’s the mechanism? Flag flip? Revert deploy? Previous image tag? Database restore? Different mechanisms for different changes.
  3. Who has the authority? Anyone on-call? A specific person? Don’t wait for approvals during an outage.
  4. What’s the time budget? “If error rate is > X for 5 minutes, we roll back.” Named, not guessed.
  5. What’s the data impact? If users wrote data through the new path, will it still make sense after rollback? If not — that’s a design problem, fix before shipping.

Reversible vs irreversible changes

Common pitfall — Changes that can’t roll back

Dropped a column, sent outgoing notifications, flipped a one-way migration. Now “roll back” isn’t an option. The deploy becomes a point of no return and your risk-tolerance must be much lower.

Better — Design reversibility in

Most changes can be made reversible with a bit of care: feature flags, expand/contract migrations, notifications that are idempotent. The cost of designing for reversibility is much lower than the cost of needing to roll back and not being able to.

⚠

Don’t romanticise “fixing forward.” In incidents, engineers often feel that rolling back is giving up — that the mature move is to fix the problem now. It’s usually the opposite. Rolling back restores known-good state; fixing forward under pressure introduces new bugs. Default to rollback; investigate the root cause after the bleeding stops.

07 · API

API Backwards Compatibility

If anyone else calls your service — another team, a mobile client, an external partner — you’re shipping an API. API changes have consequences the same day you deploy them. Make them non-breaking by default.

What counts as a breaking change

BreakingNon-breaking
Removing a field from the responseAdding a new field to the response
Renaming a fieldAdding a new endpoint
Changing a field’s typeAccepting an additional value for an enum (carefully)
Making an optional request field requiredMaking a required response field optional (usually breaking for clients that expect it)
Tightening validationLoosening validation
Changing error codes or formatsAdding new error codes (with care)

The API Change Rules

  1. Add, don’t change. Add new fields, new endpoints, new versions. Leave old ones working.
  2. Version when you must change. /v2/ exists when /v1/ still works, for some deprecation window.
  3. Announce deprecations with concrete dates, not “soon.” Give clients time to migrate.
  4. Instrument old-version usage so you know when it’s safe to remove. Don’t guess; measure.
  5. Communicate changes in writing to every known consumer. Silence breeds Slack disasters.
08 · Pre-Flight

The Pre-Flight Checklist

Every deploy has the same five questions, whether it’s a one-line fix or a major feature. Run them on yourself before you merge and before you deploy — they’re your parachute check.

The Pre-Flight Five

  1. Can I roll this back in five minutes? If no, reshape the change (flag it, split it, decompose the migration) until yes.
  2. What metric will tell me if it’s wrong? Name it before you ship. If nothing will tell you, you can’t know if it worked.
  3. Who else is affected? Downstream consumers, people on-call, scheduled jobs, backups. Think beyond your service.
  4. What’s the worst case? Data loss? User-visible outage? Revenue impact? If it’s severe, the deploy needs more guardrails or more eyes.
  5. Am I the right person to be deploying this? Experience matters. For scary changes, pair with someone who’s done this before — at least the first time.

Dos & Don’ts

DoDon’t
Deploy small changes oftenBatch two weeks of work into a Friday release
Gate new behaviour behind a flagDeploy a big change with no kill switch
Split schema changes into expand / migrate / contractRename a column in a single deploy
Write the rollback plan before the deployFigure out rollback during the incident
Roll out progressively, with named metrics at each stepFlip to 100% and hope
Add API fields instead of changing themRename a field and expect clients to adapt
Track and remove flags once they’re doneLeave the flag graveyard for the next engineer
Default to revert when an incident startsRomanticise fix-forward under pressure
🔮

The compounding payoff: a team that ships 10 times a day with flags, progressive rollout, and reversible migrations moves faster and has fewer incidents than a team that ships weekly with ceremony. Velocity and safety aren’t opposites — they’re both outputs of good deploy hygiene.

Further reading

  • Accelerate (Forsgren, Humble, Kim) — the empirical case for small, frequent deploys. Short book, strong data.
  • Continuous Delivery (Humble & Farley) — the foundational text. Older but still relevant on the principles.
  • Martin Fowler, “Feature Toggles (aka Feature Flags)” (martinfowler.com) — the reference essay on flag lifecycles and types.
  • GitLab’s database migration guidelines (docs.gitlab.com) — a rare public, concrete guide on expand/migrate/contract.
  • Gergely Orosz’s writing on deploy practices in The Pragmatic Engineer — modern, honest, pragmatic.
  • Database Reliability Engineering (Campbell & Majors) — deep on safe database operations.

One thing to try this week

  • For your next change, answer the question “if this is wrong, how do I undo it in five minutes?” in writing, before you merge.
  • Take your next scary change and reshape it to be behind a feature flag. Notice how much less worried you are shipping it.
  • Find the oldest stale flag in your codebase. Clean it up.
  • On your next DB change, split it into expand / migrate / contract — three small deploys instead of one scary one.
  • Write a rollback plan (5 lines) for your next deploy. Share it with whoever’s on-call so they know what to do if the deploy misbehaves.
