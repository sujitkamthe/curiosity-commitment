Testing Basics

Tests are not homework. They’re how you move fast without fear. Why we test, what to test, where to test, and why “I’ll add tests later” is almost never honoured.

01 · Motivation

Why We Test

Every engineer who’s been burned badly enough writes tests automatically afterwards. The good news: you don’t have to get burned. You just have to understand why tests exist, and then writing them becomes the obvious default.

Tests are not for today

When you write tests for code you just wrote, they feel redundant. You know it works — you literally just wrote it. This is the moment most engineers skip tests. It’s also the moment they shouldn’t.

Tests aren’t for today’s you. They’re for next month’s someone. Including you, after you’ve forgotten the details.

What tests actually buy you

  1. Confidence to change code. Without tests, every change is scary. With tests, refactoring is routine.
  2. Documentation of behaviour. The test suite describes what the system does, in executable form. It doesn’t rot like comments.
  3. A fast feedback loop. Tests tell you in seconds what manual QA takes hours to confirm.
  4. Design pressure. Untestable code is usually badly-designed code. Writing tests exposes the design problems early.
  5. Protection against regressions. The bug you fix today doesn’t come back next quarter — because a test now guards it.

Key Insight

The cost of tests is paid upfront, in the two hours it takes to write them. The payoff is paid in every subsequent change that doesn’t break something. The ratio is almost always in favour of writing them — which is why the engineers you admire do, and the ones who got burned enough times do.

The speed paradox

It feels like not writing tests will let you ship faster. For the first week, it does. By month three, you’re shipping slower — because every change touches code you can’t safely change, and every bug requires a full manual retest. Teams that write tests ship faster over any time horizon longer than a month. This is not an opinion — it’s repeatedly measured.

Further reading

  • Working Effectively with Legacy Code — Michael Feathers. “Legacy code is code without tests” — a definition that changes how you see the world.
  • Accelerate — Forsgren, Humble, Kim. The research on why high-performing teams test.
02 · Strategy

What to Test — The Pyramid

Not all tests are equal. They cost different amounts to write, run, and maintain. The testing pyramid is the simple mental model that tells you how much of each kind to write. Get the shape roughly right and your test suite is an asset. Get it wrong and it becomes a millstone.

The pyramid

The Three Layers

  1. Unit tests (wide base). Fast, narrow, many. A single function, class, or small unit. Run in milliseconds. You should have lots.
  2. Integration tests (middle). A few components together. Maybe talking to a real DB in a test container. Slower, fewer. One per meaningful component interaction.
  3. End-to-end tests (tiny top). Full system, click-through-the-UI, or hit-the-real-API. Slow, brittle, expensive. Only the handful of critical user flows.

The shape matters: many fast unit tests, fewer integration tests, very few e2e tests. Inverted pyramids (lots of e2e, few unit) are the #1 thing that turns a test suite into a maintenance nightmare.

What to test at each layer

LayerWhat belongs hereWhat doesn’t
UnitPure logic, calculations, validators, mappers, small algorithmsAnything that needs a real DB, network, file system
IntegrationA service + its repository + a real DB; API + framework routing; message producer/consumer wiring“Just duplicate the unit test with a real DB because it feels safer”
E2ECritical user journeys: login, checkout, main product flowEvery permutation of the UI — that’s what unit/integration covers

Scenario: the wrong-layer test

⚠

You want to check that the pricing calculator applies a 10% discount to orders over ₹1000. You write an end-to-end test: it spins up the whole system, logs in a test user, creates an order, checks the total. The test takes 45 seconds. It flakes twice a week.

Better move: the pricing rule is pure logic. Write a unit test: call calculateDiscount(order), assert the result. Runs in 2 milliseconds. Never flakes. Covers the behaviour better because you can easily test five edge cases. The e2e test, if it exists at all, confirms the happy path of the whole checkout — not the discount math.

Common pitfall — Integration testing everything

Every test spins up the full app, hits the DB, sends real emails. 1000 tests, 20-minute test run, flaky, expensive. Nobody wants to run them locally.

Better — Push logic down the pyramid

Isolate the logic. Test it as a unit. Reserve integration tests for actual integration concerns: the repository actually hits the DB correctly, the middleware actually runs.

Common pitfall — Testing the framework

You write a test that checks “if I send a POST, the framework routes it to my controller.” The framework’s job. Don’t test it.

Better — Test your logic, trust the framework

Test the controller’s behaviour given a request, not that the framework delivered the request. Framework maintainers already tested that.

03 · Design

Writing Testable Code

“This is hard to test” is almost always a design problem disguised as a testing problem. Testable code is usually just well-designed code. Learning to write it will improve your designs even when you never run the tests.

The two biggest levers

Testability Moves

  1. Pure functions where possible. Given the same input, always the same output. No global state, no side effects. Trivially testable.
  2. Dependency injection for I/O. Don’t new Clock() inside your method — take a clock parameter. Don’t fetch from the DB inside your logic — take the data in. Tests pass fakes; production passes the real thing.

Scenario: the untestable class

🛠

You’re writing an order-processing class:

class OrderProcessor {
  public void process(Order order) {
    if (LocalDateTime.now().isAfter(order.getDeadline())) {
      new EmailClient().send(order.getEmail(), "Late");
    }
    new OrderRepository().save(order);
  }
}

To test this, you have to mock the clock, mock the email client, mock the repository — all of which are newed up inside the method. Your test becomes a stack of reflection/library tricks.

The testable version:

class OrderProcessor {
  private final Clock clock;
  private final EmailClient emailClient;
  private final OrderRepository orderRepository;

  OrderProcessor(Clock clock, EmailClient e, OrderRepository r) {
    this.clock = clock;
    this.emailClient = e;
    this.orderRepository = r;
  }

  public void process(Order order) {
    if (clock.now().isAfter(order.getDeadline())) {
      emailClient.send(order.getEmail(), "Late");
    }
    orderRepository.save(order);
  }
}

Now tests pass fakes. Production passes real ones. The class is no more complex — it’s clearer about what it depends on. This is DI in its simplest form.

Pure functions are a superpower

If a function takes inputs and returns outputs without touching anything else, testing it is two lines: call, assert. That’s it. Maximise the amount of your logic that lives in pure functions. Push I/O and side effects to the edges of your program, and keep the core pure. This pattern — “functional core, imperative shell” — is widely used by engineers you admire, for exactly this reason.

Common pitfall — Logic tangled with I/O

The pricing rules are mixed in with the DB calls and the email sending. To test one rule, you have to mock the database.

Better — Separate logic from I/O

A pure calculatePricing(order, rules) function. A separate boundary layer that reads the DB, calls pricing, sends email. Logic is testable as a unit. Boundary is testable as integration.

Common pitfall — Heavy mocking

Your test is 80% mock setup. When the code changes, 20 mocks break. The mocks don’t reflect real behaviour anyway.

Better — Simple fakes or real objects

An in-memory repository is simpler than 50 lines of Mockito. A real domain object is simpler than a mock of it. Mock only things that are genuinely external and expensive.

04 · Discipline

“I’ll Add Tests Later” — The Lie You Tell Yourself

You won’t. Nobody does. “I’ll add tests later” is the single most-broken promise in engineering. Once you understand this about yourself, you can choose differently: either accept you’re shipping without tests (and its consequences), or write them now.

Why “later” doesn’t happen

  • The code is already merged. Nothing’s broken. Why are you writing tests for working code?
  • You’ve context-switched. Next sprint, the details you needed for the test are gone.
  • There’s always a next story. The backlog never runs out.
  • The code is already structured in a hard-to-test way — the time to think about testability was before, not after.
  • Nobody is measuring. Once it’s shipped, the social pressure disappears.

Key Insight

The test you don’t write today is not a test you’ll write next week. It’s a test that will never exist. Plan accordingly — either write it now, or accept that this code will be harder to change forever.

The case for writing tests with the code

Tests written alongside the code are cheap. You’re already loaded with context: the inputs, the edge cases, the gotchas. Writing two tests takes ten minutes. Writing the same two tests three weeks later — when you’ve forgotten all of that — takes forty minutes and they’ll be worse tests because you’ll only remember the happy path.

The Minimum Test Rule

  1. Every MR ships with tests for the new behaviour.
  2. Every bug fix ships with a regression test — the test that would have caught the bug.
  3. No “tests coming in a follow-up” — unless there’s an explicit, scheduled ticket AND a test for the new behaviour is actually present.
⚠

The honest alternative: if you genuinely can’t write tests (legacy code, time pressure, whatever), say so out loud. “This MR has no tests because X. I’ll open a follow-up ticket for test coverage.” Then actually do the follow-up. Silently shipping untested code is the failure mode — not the decision to occasionally do it with good reason.

Dos & Don’ts

DoDon’t
Write a test with the code that needs itWrite code, ship, “tests next sprint”
When a bug is fixed, write the test that would have caught itFix the bug, move on, leave the regression possibility wide open
If you skip tests, say so in the MR description — explicitlyQuietly ship untested code
Treat tests as part of “done”, not a bonusTreat tests as homework to do if time allows
05 · Technique

TDD When It Helps

Test-Driven Development isn’t dogma and isn’t a religion. It’s a technique: write a failing test, write the minimum code to pass, refactor, repeat. For some kinds of work it’s tremendously helpful. For others, it’s overkill. Knowing when to use it is part of the skill.

When TDD genuinely helps

Pure logic with clear rules

Calculators, validators, parsers, state machines. The inputs and outputs are knowable. TDD gives you steady forward progress.

Bug reproduction

Write a failing test that captures the bug. Now you have a concrete target. Fix until green. You now also have a regression test — free.

Legacy code you’re afraid to change

Add characterisation tests first — tests that capture existing behaviour. Now you can refactor with confidence.

When TDD is not the right tool

Exploratory / spike work

You don’t know what you’re building yet. Tests calcify decisions you haven’t made. Spike first; add tests after you know the shape.

UI-heavy work

Visual, interactive, design-iterative work. TDD for the logic underneath; dogfood and e2e test for the surface.

Integration glue

Wiring framework components together. The “test” is often the integration test itself; unit TDD doesn’t add much.

The TDD Cycle (when you use it)

  1. Red: write the simplest failing test for the next tiny bit of behaviour you want.
  2. Green: write the simplest code that passes. Not clean — just passing.
  3. Refactor: clean up both test and code, with the safety net of the test still passing.
  4. Repeat. Small cycles. One assertion at a time.
💡

You don’t have to TDD everything to get 80% of the benefit. Try it for your next pure-logic task — a function that maps inputs to outputs. You’ll feel the value immediately. You’ll also learn why people who swear by TDD sound like they’re in a cult: when the fit is right, the flow is unreasonably nice.

Further reading

  • Test-Driven Development: By Example — Kent Beck. The canonical book; short and practical.
  • Growing Object-Oriented Software, Guided by Tests — Freeman & Pryce. The deeper book on how tests shape design.
06 · Coverage

Testing the Edges, Not Just the Happy Path

A test suite that covers only the happy path is a test suite that ships bugs. The bugs you care about almost always live at the edges — empty inputs, boundary values, unexpected types, failure modes. Thinking through the edges is often where the best part of writing tests happens.

The edges to think about

The Edge Checklist

  1. Empty. Empty list, empty string, empty map, null/None/nil.
  2. One. Collections of size one — often break algorithms designed for “many.”
  3. Many. Large inputs — performance edges, off-by-ones in pagination, memory.
  4. Boundary values. Just below, at, just above a threshold. The classic bug site.
  5. Invalid input. Wrong type, malformed, impossible combination.
  6. Failure modes. DB down, network timeout, third-party 500. What does your code do?
  7. Concurrent access. Two requests hit the same resource at once. Race conditions.
  8. Time boundaries. Timezone edges, daylight-savings, leap days, year rollover.

Scenario: the day-one pagination bug

🛠

You ship pagination. Happy path test: 100 items, 10 per page — 10 pages. Works. Ship.

Production: someone has 0 items. Your code divides by something that becomes 0. Or returns a blank page with “Page 1 of 0.” Or worse, throws.

Edges you should have tested: 0 items, 1 item, exactly N items (the page boundary), N+1 items (one over), a fractional page (113 items, 10/page = 11.3 pages). Each is one extra test. Each catches a different real bug.

Common pitfall — One happy-path test per story

You wrote a test. You feel done. The test covers the path you manually clicked through. No edges.

Better — At least one happy path, one boundary, one failure

Three tests minimum, usually. It takes ten extra minutes. The edges are where the bugs hide.

Common pitfall — Only testing “success returns”

You verify the function returns the right thing on success. You never check what it does when something goes wrong. Production finds out for you.

Better — Test the failure paths

What does the code do when the DB is down, the third party returns 500, input is malformed? That’s often where the actual bug is.

Key Insight

Five happy-path tests cover less ground than one happy-path plus four edge tests. When you finish writing a test, pause and ask: “what’s the weirdest input this could get?” Write one more test.

Further reading

  • Unit Testing Principles, Practices and Patterns — Vladimir Khorikov. The book on testable design after you know the basics.
  • The Art of Unit Testing — Roy Osherove. Approachable introduction with real examples.
  • Martin Fowler on testing (martinfowler.com) — particularly his pieces on the testing pyramid and test doubles.

One thing to try this week

  • For your next story, write the tests before calling it done — not in a follow-up.
  • For your next bug fix, write the regression test first.
  • On your next test, add one edge case you wouldn’t normally think of.
  • Try TDD on one small pure-logic task. See how it feels.
  • Refactor one function to take its dependencies as parameters instead of constructing them internally.
